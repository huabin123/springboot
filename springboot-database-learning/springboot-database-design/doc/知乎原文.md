# MySQL面试核心脉络

MySQL面试最坑的地方，不是题难，是特别碎。

今天问你B+树+，明天问你MVCC+,下一分钟又跳到redo log、死锁、explain、深分页。很多人准备到最后，脑子里全是名词，真到面试现场一追问场景，立刻开始打结。

其实MySQL面试没那么散。翻来覆去，面试官就想看四件事：数据怎么找，并发下怎么错，宕机后怎么活过来，一条SQL为什么会慢。

把这四件事顺明白，80%的MySQL面试题都能接住。

## 第一大块：索引

先说最容易暴露功底的索引。

很多人一上来就背"最左前缀""索引失效""覆盖索引"，背得像口诀。面试官真正想听的不是这些词，是你知不知道索引为什么会快。最短的答案就是：InnoDB的聚簇索引+和普通二级索引，底层主流结构都是B+树。值钱的不是"有树"，而是"有序"。等值查询、范围查询、排序、前缀匹配，本质上都在利用这个有序性。

所以常见题目都会从这里长出来。为什么InnoDB不用红黑树做索引，要用B+树。为什么联合索引要满足最左前缀。为什么Like'abc%'可能走索引，1ike'%abc'大概率不走。为什么字段上套函数、隐式类型转换、0、！=经常把索引打废。什么叫回表，什么叫覆盖索引，覆盖索引为什么能快。

**📚 相关文档**:
- [为什么InnoDB用B+树而不用红黑树](./02-索引优化/01-为什么InnoDB用B+树而不用红黑树.md) - B+树原理详解
- [为什么有最左匹配原则](./02-索引优化/02-为什么有最左匹配原则.md) - 联合索引原理
- [唯一索引和普通索引的写入读取区别](./02-索引优化/05-唯一索引和普通索引的写入读取区别.md) - 回表和覆盖索引

这种题别答成定义大全，最好直接顺着底层往下说。

比如联合索引(a,b,c),为什么where a=1andc=3往往用不满索引？不是数据库脾气怪，是因为B+树排的是(a,b,c)这个整体顺序。a=1能先缩小一段范围，但中间的b没定住，c就没法继续在树上缩小范围。你把这件事讲明白，比背"最左前缀原则"有用得多。

## 第二大块：事务和MVCC

第二大块一定是事务和MVCC。

这块最容易答成背课文。四个隔离级别大家都会背，真拉开差距的是你知不知道noDB里的"读"和"写"根本不是一个模型。

普通select在Read Committed和Repeatable Read下，大多是快照读，底下靠的是undo log版本链加Read View。

select·for update、update、delete这些是当前读，会真正去碰锁。所以这一组高频题基本绕不开这些：MVCC怎么实现，Read View里有什么，为什么MySQL默认是Repeatable Read,快照读和当前读的区别是什么，Gap Lock+和Next-Key Lock+是干嘛的，为什么同样是for update,走索引和不走索引时锁范围会完全不一样。

- [快照读和当前读的区别](./02-索引优化/06-快照读和当前读的区别.md) - 两种读模式的区别

- 比如Repeatable Read不是已经解决幻读了吗，为什么还有Gap Lock。

- 比如update明明说是行锁，为什么一个条件没走索引，最后看起来像把半张表都锁住了。
  - 📖 [MySQL锁机制](./03-事务与并发/01-MySQL锁机制.md) - 锁的范围和访问路径

这些问题只靠背定义接不住，因为它们问的不是名词，是执行过程。

## 第三大块：锁

第三大块是锁，而且锁这块不能单独背，它和索引绑死了。

很多人把"行锁""间隙锁""死锁"单独列成一章，这样准备最容易崩。InnoDB说是行锁，实际锁的是索引记录。访问路径不对，行锁就会锁出你完全不想看到的范围。

所以常见题会长这样：

- select.:,for update到底锁了什么。
- 为什么一个update会把另一个看起来不相关的语句堵住。
- 什么情况下会发生死锁。
- MySQL发现死锁之后怎么处理。
- 为什么两个事务都只是改两行数据，也能改出死锁。

- [有了间隙锁为什么还要有临键锁？](./06-问题解答/问题03-有了间隙锁为什么还要有临键锁？.md)
- [什么隔离级别、什么场景会使用间隙锁](./06-问题解答/问题02-那什么隔离级别，什么场景会使用间隙锁呢？.md)

这一块最值钱的一句话其实很简单：InnoDB的并发控制从来不是"你以为锁住一行就真的只锁一行"，而是"你的访问路径决定它最终锁了什么"。

这句话能把很多碎问题串起来。

## 第四大块：日志和恢复

第四大块是日志和恢复。

这一块一堆人背得最乱。redo log、undo log、binlog,三个词都见过，真让他说清楚各自干什么，经常越讲越糊。

最顺的讲法不是背定义，而是按一次事务提交的时间线讲。

数据页先改在Buffer Pool+里，不是每次改完都立刻刷盘。要保证宕机后这些修改不会丢，就得把"页被怎么改过"记到redo log。

要支持回滚和MVCC,就得保留I旧版本到undo log。

如果还要做主从复制、归档、CDC,就得再记逻辑变更到binlog。

然后问题自然就来了：

- 既然有redo log,为什么还要binlog。
- redo log和binlog为什么要两阶段提交。
- 数据库crash之后靠什么恢复。
- 主从复制看的到底是哪份日志。
- checkpoint是干嘛的，redo log写满了会怎么样。

**📚 相关文档**:
- [Redo和Undo日志](./03-事务与并发/03-Redo和Undo日志.md) - 三种日志详解、两阶段提交、崩溃恢复

你能把这一条线讲顺，MySQL这块基本就不是"只会用"的水平了。

## 第五大块：慢SQL和EXPLAIN

最后一大块是慢SQL和EXPLAIN。

这块最能拉开"背八股"和"真干过活"的差距。

高频问题非常固定：explain重点看什么，type到什么级别算差，rows和Extra怎么看，using temporary、using filesort代表什么，为什么深分页慢，为什么有索引还是慢，为什么join一多就容易炸。

这一块千万别答成"优化技巧大全"。面试官真正想听的是你有没有一套排查顺序。

- 看到ALL,先怀疑访问路径不对。
- 看到using temporary、using filesort,先盯排序、分组和联合索引。
- 看到1imit100000,20，先别怪优化器，先想是不是业务设计就不该这么翻。
- 看到函数、隐式转换、不同类型的joi,先怀疑索引根本没用起来。

**📚 相关文档**:
- [慢SQL排查与EXPLAIN详解](./05-性能优化/08-慢SQL排查与EXPLAIN详解.md) - EXPLAIN字段详解、慢SQL排查5步法、索引失效场景、深分页优化

如果再往上走一点，面试官会开始问那些特别烦但特别真实的问题。

- 联合索引明明建了，为什么order by还是filesort。
- 条件都命中了索引，为什么还是慢。
- 主键用自增还是UUD。
- count(\*)和count(1)到底谁快。

这些题表面不一样，底下还是那几件事：访问路径、回表次数、页分裂、局部性、锁范围。像count(\*)和count(1)这种题，真拉开差距的也从来不是死背结论，而是你知不知道InnoDB下多数时候瓶颈在访问路径，不在\*还是1这个字面形式。
