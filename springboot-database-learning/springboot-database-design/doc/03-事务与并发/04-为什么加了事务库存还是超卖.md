# 为什么加了事务，库存还是能超卖？

## 一、快速结论

**加了事务不等于加了锁！超卖的本质是"先查后改"的并发竞态条件。**

| 场景 | 是否超卖 | 原因 |
|------|---------|------|
| `@Transactional` + 普通查询 | ✅ 会超卖 | 快照读，不加锁 |
| `@Transactional` + `SELECT FOR UPDATE` | ❌ 不超卖 | 当前读，加排他锁 |
| `@Transactional` + 直接UPDATE | ❌ 不超卖 | UPDATE自动加锁 |
| 乐观锁（version字段） | ❌ 不超卖 | CAS机制 |

## 二、超卖场景复现

### 2.1 典型的超卖代码

```java
@Service
public class OrderService {
    
    @Autowired
    private ProductMapper productMapper;
    
    @Autowired
    private OrderMapper orderMapper;
    
    // ❌ 错误示例：会超卖
    @Transactional
    public void createOrder(Long productId, Integer quantity) {
        // 1. 查询库存
        Product product = productMapper.selectById(productId);
        
        // 2. 判断库存
        if (product.getStock() < quantity) {
            throw new RuntimeException("库存不足");
        }
        
        // 3. 扣减库存
        product.setStock(product.getStock() - quantity);
        productMapper.updateById(product);
        
        // 4. 创建订单
        Order order = new Order();
        order.setProductId(productId);
        order.setQuantity(quantity);
        orderMapper.insert(order);
    }
}
```

### 2.2 超卖的执行时序

```
假设：商品库存 = 1，两个用户同时下单

时间线：
T1: 事务A开始
T2: 事务B开始
T3: 事务A查询库存 → stock = 1  ✅ 库存充足
T4: 事务B查询库存 → stock = 1  ✅ 库存充足（快照读，读到旧值）
T5: 事务A扣减库存 → stock = 0
T6: 事务A提交
T7: 事务B扣减库存 → stock = -1  ❌ 超卖了！
T8: 事务B提交

结果：库存变成 -1，超卖了！
```

### 2.3 为什么会这样？

```
核心问题：普通SELECT是"快照读"，不加锁

事务A：SELECT * FROM product WHERE id = 1;
       ↓
       读取快照（MVCC）
       
事务B：SELECT * FROM product WHERE id = 1;
       ↓
       读取快照（MVCC）
       
两个事务读到的都是同一个快照，都认为库存充足！
```

## 三、解决方案

### 3.1 方案1：SELECT FOR UPDATE（悲观锁）

```java
// ✅ 正确示例：不会超卖
@Transactional
public void createOrder(Long productId, Integer quantity) {
    // 1. 查询库存（加排他锁）
    Product product = productMapper.selectByIdForUpdate(productId);
    
    // 2. 判断库存
    if (product.getStock() < quantity) {
        throw new RuntimeException("库存不足");
    }
    
    // 3. 扣减库存
    product.setStock(product.getStock() - quantity);
    productMapper.updateById(product);
    
    // 4. 创建订单
    Order order = new Order();
    order.setProductId(productId);
    order.setQuantity(quantity);
    orderMapper.insert(order);
}
```

**Mapper定义**：
```java
@Select("SELECT * FROM product WHERE id = #{id} FOR UPDATE")
Product selectByIdForUpdate(Long id);
```

**执行时序**：
```
时间线：
T1: 事务A开始
T2: 事务B开始
T3: 事务A查询库存 FOR UPDATE → 获得锁，stock = 1
T4: 事务B查询库存 FOR UPDATE → 等待锁...
T5: 事务A扣减库存 → stock = 0
T6: 事务A提交 → 释放锁
T7: 事务B获得锁，查询库存 → stock = 0  ❌ 库存不足
T8: 事务B抛出异常，回滚

结果：库存 = 0，没有超卖！
```

### 3.2 方案2：直接UPDATE（推荐）

```java
// ✅ 最佳实践：直接UPDATE，不需要先查询
@Transactional
public void createOrder(Long productId, Integer quantity) {
    // 1. 直接扣减库存（带库存检查）
    int rows = productMapper.decreaseStock(productId, quantity);
    
    // 2. 判断是否扣减成功
    if (rows == 0) {
        throw new RuntimeException("库存不足");
    }
    
    // 3. 创建订单
    Order order = new Order();
    order.setProductId(productId);
    order.setQuantity(quantity);
    orderMapper.insert(order);
}
```

**Mapper定义**：
```java
@Update("UPDATE product SET stock = stock - #{quantity} " +
        "WHERE id = #{productId} AND stock >= #{quantity}")
int decreaseStock(@Param("productId") Long productId, 
                  @Param("quantity") Integer quantity);
```

**为什么这样不会超卖？**
```
UPDATE语句自动加排他锁！

时间线：
T1: 事务A执行 UPDATE → 获得锁，stock = 1 - 1 = 0，影响行数 = 1
T2: 事务B执行 UPDATE → 等待锁...
T3: 事务A提交 → 释放锁
T4: 事务B获得锁，执行 UPDATE → stock = 0，不满足 stock >= 1，影响行数 = 0
T5: 事务B抛出异常，回滚

结果：库存 = 0，没有超卖！
```

### 3.3 方案3：乐观锁（version字段）

```java
// 表结构
CREATE TABLE product (
    id BIGINT PRIMARY KEY,
    name VARCHAR(100),
    stock INT,
    version INT DEFAULT 0,  -- 版本号
    INDEX idx_version (version)
);
```

```java
// ✅ 乐观锁方案
@Transactional
public void createOrder(Long productId, Integer quantity) {
    int maxRetry = 3;  // 最多重试3次
    
    for (int i = 0; i < maxRetry; i++) {
        // 1. 查询库存和版本号
        Product product = productMapper.selectById(productId);
        
        // 2. 判断库存
        if (product.getStock() < quantity) {
            throw new RuntimeException("库存不足");
        }
        
        // 3. 扣减库存（带版本号检查）
        int rows = productMapper.decreaseStockWithVersion(
            productId, 
            quantity, 
            product.getVersion()
        );
        
        // 4. 如果更新成功，创建订单
        if (rows > 0) {
            Order order = new Order();
            order.setProductId(productId);
            order.setQuantity(quantity);
            orderMapper.insert(order);
            return;  // 成功
        }
        
        // 5. 更新失败，重试
        Thread.sleep(10);  // 短暂等待
    }
    
    throw new RuntimeException("系统繁忙，请稍后重试");
}
```

**Mapper定义**：
```java
@Update("UPDATE product " +
        "SET stock = stock - #{quantity}, version = version + 1 " +
        "WHERE id = #{productId} " +
        "AND stock >= #{quantity} " +
        "AND version = #{version}")
int decreaseStockWithVersion(@Param("productId") Long productId,
                              @Param("quantity") Integer quantity,
                              @Param("version") Integer version);
```

**执行时序**：
```
时间线：
T1: 事务A查询 → stock = 1, version = 0
T2: 事务B查询 → stock = 1, version = 0
T3: 事务A更新 → WHERE version = 0，成功，version = 1
T4: 事务A提交
T5: 事务B更新 → WHERE version = 0，失败（version已经是1了）
T6: 事务B重试 → 查询 stock = 0, version = 1 → 库存不足

结果：库存 = 0，没有超卖！
```

### 3.4 方案4：分布式锁（Redis/Redisson）

```java
@Autowired
private RedissonClient redissonClient;

// ✅ 分布式锁方案
@Transactional
public void createOrder(Long productId, Integer quantity) {
    String lockKey = "product_lock:" + productId;
    RLock lock = redissonClient.getLock(lockKey);
    
    try {
        // 1. 尝试获取锁（等待10秒，锁30秒自动释放）
        boolean locked = lock.tryLock(10, 30, TimeUnit.SECONDS);
        if (!locked) {
            throw new RuntimeException("系统繁忙，请稍后重试");
        }
        
        // 2. 查询库存
        Product product = productMapper.selectById(productId);
        
        // 3. 判断库存
        if (product.getStock() < quantity) {
            throw new RuntimeException("库存不足");
        }
        
        // 4. 扣减库存
        product.setStock(product.getStock() - quantity);
        productMapper.updateById(product);
        
        // 5. 创建订单
        Order order = new Order();
        order.setProductId(productId);
        order.setQuantity(quantity);
        orderMapper.insert(order);
        
    } finally {
        // 6. 释放锁
        if (lock.isHeldByCurrentThread()) {
            lock.unlock();
        }
    }
}
```

## 四、方案对比

### 4.1 性能对比

| 方案 | 并发性能 | 实现复杂度 | 适用场景 |
|------|---------|-----------|---------|
| **直接UPDATE** | ⭐⭐⭐⭐⭐ | ⭐ | 推荐，简单高效 |
| **SELECT FOR UPDATE** | ⭐⭐⭐ | ⭐⭐ | 需要先查询再判断 |
| **乐观锁** | ⭐⭐⭐⭐ | ⭐⭐⭐ | 冲突少的场景 |
| **分布式锁** | ⭐⭐ | ⭐⭐⭐⭐ | 分布式系统 |

### 4.2 详细对比

```
1. 直接UPDATE（推荐）
   优点：
   - 性能最好（一条SQL搞定）
   - 代码最简单
   - 不会死锁
   
   缺点：
   - 无法在扣减前做复杂判断
   
   适用：90%的场景

2. SELECT FOR UPDATE
   优点：
   - 可以先查询再判断
   - 逻辑清晰
   
   缺点：
   - 需要两条SQL
   - 可能死锁
   - 锁持有时间长
   
   适用：需要复杂判断的场景

3. 乐观锁
   优点：
   - 不加数据库锁
   - 并发性能好
   
   缺点：
   - 需要重试机制
   - 冲突多时性能差
   - 需要额外字段
   
   适用：冲突少的场景（如秒杀）

4. 分布式锁
   优点：
   - 跨数据库、跨服务
   
   缺点：
   - 依赖Redis
   - 性能最差
   - 实现复杂
   
   适用：分布式系统
```

## 五、常见误区

### 误区1：加了@Transactional就不会超卖

```java
// ❌ 错误理解
@Transactional  // 只保证原子性，不保证隔离性！
public void createOrder(Long productId, Integer quantity) {
    Product product = productMapper.selectById(productId);  // 快照读，不加锁
    if (product.getStock() < quantity) {
        throw new RuntimeException("库存不足");
    }
    product.setStock(product.getStock() - quantity);
    productMapper.updateById(product);
}

// ✅ 正确理解
@Transactional只保证：
- 原子性：要么全成功，要么全失败
- 不保证：并发情况下的数据一致性

要防止超卖，必须加锁或用乐观锁！
```

### 误区2：隔离级别设置成SERIALIZABLE就不会超卖

```java
// ❌ 错误做法
@Transactional(isolation = Isolation.SERIALIZABLE)
public void createOrder(Long productId, Integer quantity) {
    Product product = productMapper.selectById(productId);
    // ...
}

问题：
1. SERIALIZABLE性能极差（串行执行）
2. 可能导致大量锁等待和死锁
3. 不是解决超卖的正确方式

正确做法：用REPEATABLE_READ + FOR UPDATE
```

### 误区3：UPDATE一定不会超卖

```java
// ❌ 错误示例
@Update("UPDATE product SET stock = stock - #{quantity} WHERE id = #{productId}")
int decreaseStock(@Param("productId") Long productId, 
                  @Param("quantity") Integer quantity);

问题：没有检查库存是否充足，可能扣成负数！

// ✅ 正确示例
@Update("UPDATE product SET stock = stock - #{quantity} " +
        "WHERE id = #{productId} AND stock >= #{quantity}")
int decreaseStock(@Param("productId") Long productId, 
                  @Param("quantity") Integer quantity);
```

## 六、面试高频问题

### Q1：为什么加了@Transactional还是会超卖？

**标准答案**：
```
@Transactional只保证原子性，不保证并发安全。

超卖的根本原因是"先查后改"的竞态条件：
1. 两个事务同时查询库存（快照读，不加锁）
2. 都读到库存充足
3. 都执行扣减操作
4. 结果超卖

解决方案：
1. 使用 SELECT FOR UPDATE（悲观锁）
2. 直接UPDATE带库存检查（推荐）
3. 使用乐观锁（version字段）
```

### Q2：SELECT FOR UPDATE 和 直接UPDATE 哪个好？

**标准答案**：
```
推荐直接UPDATE，原因：

1. 性能更好：
   - 直接UPDATE：1条SQL
   - SELECT FOR UPDATE：2条SQL

2. 代码更简单：
   - 直接UPDATE：一行代码
   - SELECT FOR UPDATE：需要先查询再更新

3. 不会死锁：
   - 直接UPDATE：锁持有时间短
   - SELECT FOR UPDATE：锁持有时间长，容易死锁

什么时候用SELECT FOR UPDATE？
- 需要在扣减前做复杂判断
- 需要读取多个字段进行计算
```

### Q3：乐观锁和悲观锁怎么选？

**标准答案**：
```
看冲突概率：

1. 冲突少（<10%）：用乐观锁
   - 例如：普通商品下单
   - 优点：并发性能好
   - 缺点：冲突时需要重试

2. 冲突多（>50%）：用悲观锁
   - 例如：热门商品秒杀
   - 优点：不需要重试
   - 缺点：并发性能差

3. 最佳实践：
   - 普通场景：直接UPDATE（悲观锁）
   - 秒杀场景：Redis预扣减 + 异步入库
```

### Q4：分布式系统怎么防止超卖？

**标准答案**：
```
方案1：分布式锁（Redisson）
- 适用：多个服务操作同一个数据库
- 缺点：依赖Redis，性能差

方案2：数据库锁（推荐）
- 直接UPDATE带库存检查
- 不需要额外组件
- 性能好

方案3：消息队列串行化
- 所有扣库存请求进队列
- 单线程消费
- 适用：对实时性要求不高的场景

方案4：Redis预扣减
- 库存放Redis
- 先在Redis扣减
- 异步同步到数据库
- 适用：秒杀场景
```

## 八、总结

### 核心要点

```
为什么加了事务还是会超卖？

1. 事务不等于锁：
   - @Transactional只保证原子性
   - 不保证并发安全

2. 快照读不加锁：
   - 普通SELECT是快照读
   - 两个事务读到同一个快照
   - 都认为库存充足

3. 解决方案：
   - 直接UPDATE（推荐）
   - SELECT FOR UPDATE
   - 乐观锁
   - 分布式锁
```

### 记忆口诀

```
事务保原子不保并发，快照读取不会加锁
先查后改竞态条件，两个事务同时扣减
直接UPDATE带库存检查，一条SQL解决问题
FOR UPDATE悲观加锁，乐观锁用版本号
```

**最终答案：@Transactional不加锁，普通SELECT是快照读，"先查后改"导致超卖，用直接UPDATE解决。**
