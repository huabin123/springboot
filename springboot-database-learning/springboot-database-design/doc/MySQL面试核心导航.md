## 📚 主线一：数据怎么找（索引）

### 核心问题
**为什么InnoDB用B+树，不用红黑树？**

**一句话答案**：减少磁盘IO次数。

```
红黑树：100万数据，树高20层，20次IO
B+树：  100万数据，树高3层，3次IO

B+树优势：
1. 树高低 → IO少
2. 叶子节点有序链表 → 范围查询快
3. 节点大小=磁盘页 → 一次IO读更多key
```

### 高频考点

#### 1. B+树 vs 其他数据结构
```
Q: 为什么不用哈希表？
A: 不支持范围查询、排序、最左前缀匹配

Q: 为什么不用B树？
A: B+树非叶子节点不存数据，能存更多key，树更矮

Q: 为什么不用跳表？
A: 跳表是概率平衡，磁盘IO场景下B+树更优
```

#### 2. 联合索引最左匹配
```
索引：INDEX idx_abc (a, b, c)

✅ 能用索引：
WHERE a = 1
WHERE a = 1 AND b = 2
WHERE a = 1 AND b = 2 AND c = 3

❌ 不能用索引：
WHERE b = 2
WHERE c = 3
WHERE b = 2 AND c = 3

⚠️ 部分用索引：
WHERE a = 1 AND c = 3  -- 只用到a
WHERE a = 1 AND b > 2 AND c = 3  -- 只用到a和b
```

#### 3. 索引失效场景
```
1. 字段上有函数：WHERE YEAR(create_time) = 2024
2. 隐式类型转换：WHERE phone = 13812345678（phone是VARCHAR）
3. LIKE以%开头：WHERE name LIKE '%张三'
4. OR条件有字段没索引
5. !=、<>、NOT IN
```

#### 4. 回表 vs 覆盖索引
```
回表：
- 二级索引 → 得到主键 → 聚簇索引 → 得到完整数据
- 需要两次查询

覆盖索引：
- 查询的字段都在索引中
- 不需要回表
- Extra: Using index
```

## 📚 主线二：并发下怎么不错（事务与锁）

### 核心问题
**为什么加了事务，库存还是能超卖？**

**一句话答案**：事务不等于锁，普通SELECT是快照读，不加锁。

```
超卖原因：
1. 两个事务同时查询库存（快照读）
2. 都读到库存充足
3. 都执行扣减
4. 结果超卖

解决方案：
1. SELECT FOR UPDATE（悲观锁）
2. 直接UPDATE带库存检查（推荐）
3. 乐观锁（version字段）
```

### 高频考点

#### 1. 快照读 vs 当前读
```
快照读（不加锁）：
- 普通SELECT
- 读取MVCC版本链
- 不会阻塞其他事务

当前读（加锁）：
- SELECT FOR UPDATE
- SELECT LOCK IN SHARE MODE
- UPDATE、DELETE、INSERT
- 读取最新版本
- 会加锁
```

#### 2. MVCC原理
```
核心组件：
1. 隐藏字段：DB_TRX_ID（事务ID）、DB_ROLL_PTR（回滚指针）
2. Undo Log：版本链
3. Read View：可见性判断

可重复读（RR）：
- 事务开始时创建Read View
- 整个事务期间使用同一个Read View

读已提交（RC）：
- 每次查询都创建新的Read View
```

#### 3. 锁的类型
```
按粒度：
- 表锁：开销小，并发低
- 行锁：开销大，并发高

按类型：
- 共享锁（S锁）：读锁，可以并发
- 排他锁（X锁）：写锁，互斥

InnoDB行锁：
- 记录锁（Record Lock）：锁定单行
- 间隙锁（Gap Lock）：锁定间隙，防止插入
- 临键锁（Next-Key Lock）：记录锁+间隙锁
```

#### 4. 锁的范围
```
关键点：InnoDB锁的是索引记录，不是行！

走主键索引：
SELECT * FROM user WHERE id = 1 FOR UPDATE;
→ 只锁id=1这一行

走普通索引：
SELECT * FROM user WHERE age = 25 FOR UPDATE;
→ 锁age=25的所有行 + 间隙

不走索引：
SELECT * FROM user WHERE name = '张三' FOR UPDATE;
→ 锁全表！
```

#### 5. 死锁
```
原因：
- 两个事务互相等待对方持有的锁

避免：
1. 按相同顺序访问资源
2. 缩小事务范围
3. 降低隔离级别
4. 添加合理索引（避免全表锁）
```

## 📚 主线三：宕机后怎么活（日志与恢复）

### 核心问题
**redo log、undo log、binlog各自干什么？**

**一句话答案**：按事务提交的时间线理解。

```
事务提交流程：

1. 修改数据（在Buffer Pool）
   ↓
2. 记录Undo Log（支持回滚和MVCC）
   ↓
3. 记录Redo Log（保证持久性）
   ↓
4. 记录Binlog（主从复制、归档）
   ↓
5. 提交事务（两阶段提交）
```

### 必读文档
- [Redo和Undo日志](./03-事务与并发/03-Redo和Undo日志.md) ⭐⭐⭐⭐⭐
- [MVCC机制](./03-事务与并发/02-MVCC机制.md) ⭐⭐⭐⭐

### 高频考点

#### 1. 三种日志的作用
```
Redo Log（重做日志）：
- 作用：保证持久性（Durability）
- 内容：物理日志，记录"在某个数据页上做了什么修改"
- 时机：事务提交前写入
- 场景：数据库crash后恢复

Undo Log（回滚日志）：
- 作用：支持回滚和MVCC
- 内容：逻辑日志，记录"相反的操作"
- 时机：修改数据前写入
- 场景：事务回滚、快照读

Binlog（归档日志）：
- 作用：主从复制、数据恢复
- 内容：逻辑日志，记录"SQL语句"或"行变更"
- 时机：事务提交时写入
- 场景：主从同步、数据归档
```

#### 2. 为什么需要两阶段提交？
```
问题：Redo Log和Binlog必须保持一致

如果不用两阶段提交：
1. 先写Redo Log，再写Binlog
   → 如果Binlog写入失败，主库有数据，从库没有

2. 先写Binlog，再写Redo Log
   → 如果Redo Log写入失败，从库有数据，主库crash后没有

两阶段提交：
1. Prepare阶段：写Redo Log，状态为prepare
2. Commit阶段：写Binlog，然后Redo Log状态改为commit

崩溃恢复：
- Redo Log为prepare，Binlog完整 → 提交
- Redo Log为prepare，Binlog不完整 → 回滚
```

#### 3. Redo Log写满了会怎样？
```
Redo Log是循环写：
- 默认2个文件，每个1GB
- 写满后从头开始覆盖

写满的后果：
- 阻塞所有更新操作
- 等待checkpoint推进
- 刷脏页到磁盘

优化：
- 增大innodb_log_file_size
- 增大innodb_log_files_in_group
```

## 📚 主线四：为什么会慢（执行计划）

### 核心问题
**一条SQL明明建了索引，为什么还是慢？**

**一句话答案**：看访问路径，不是看有没有索引。

```
慢SQL排查5步法：

1. EXPLAIN看type
   - ALL、index → 全表扫描

2. 看key
   - NULL → 索引没用上

3. 看rows
   - 太大 → 索引不够精确

4. 看Extra
   - Using filesort → 需要排序
   - Using temporary → 使用临时表

5. 看实际执行时间
   - SHOW PROFILES
```

### 必读文档
- [慢SQL排查与EXPLAIN详解](./05-性能优化/08-慢SQL排查与EXPLAIN详解.md) ⭐⭐⭐⭐⭐
- [长事务优化指南](./05-性能优化/07-长事务优化指南.md) ⭐⭐⭐⭐

### 高频考点

#### 1. EXPLAIN核心字段
```
type（访问类型）：
system > const > eq_ref > ref > range > index > ALL

key（使用的索引）：
- NULL：没用索引
- 有值：用了索引

rows（扫描行数）：
- 越少越好

Extra（额外信息）：
- Using filesort：需要排序，慢
- Using temporary：使用临时表，慢
- Using index：覆盖索引，快
- Using index condition：索引下推，好
```

#### 2. 索引失效场景
```
1. 字段上有函数：
   WHERE YEAR(create_time) = 2024

2. 隐式类型转换：
   WHERE phone = 13812345678（phone是VARCHAR）

3. LIKE以%开头：
   WHERE name LIKE '%张三'

4. OR条件有字段没索引

5. !=、<>、NOT IN
```

#### 3. 深分页优化
```
问题：
SELECT * FROM user LIMIT 1000000, 20;
→ 需要扫描1000020行

优化方案1：子查询
SELECT * FROM user 
WHERE id >= (SELECT id FROM user LIMIT 1000000, 1)
LIMIT 20;

优化方案2：记录上次最大ID
SELECT * FROM user WHERE id > 上次最大ID LIMIT 20;

优化方案3：延迟关联
SELECT * FROM user u
JOIN (SELECT id FROM user LIMIT 1000000, 20) t
ON u.id = t.id;
```

#### 4. JOIN优化
```
问题：
- JOIN字段没有索引
- JOIN字段类型不一致

优化：
1. 给JOIN字段加索引
2. 确保JOIN字段类型一致
3. 小表驱动大表
```

## 🎯 四大核心问题（必须会）

### 1. 为什么InnoDB用B+树，不用红黑树？
```
核心：减少磁盘IO次数

1. 树高差距：
   - 红黑树：20层，20次IO
   - B+树：3层，3次IO

2. 范围查询：
   - 红黑树：每个值都要从根查找
   - B+树：叶子节点有序链表，顺序扫描

3. 磁盘友好：
   - B+树节点大小=磁盘页（16KB）
   - 一次IO读一个节点，包含上千个key
```
👉 [详细文档](./02-索引优化/01-为什么InnoDB用B+树而不用红黑树.md)

### 2. 为什么加了事务，库存还是能超卖？
```
核心：事务不等于锁

1. 超卖原因：
   - 普通SELECT是快照读，不加锁
   - 两个事务读到同一个快照
   - 都认为库存充足

2. 解决方案：
   - SELECT FOR UPDATE（悲观锁）
   - 直接UPDATE带库存检查（推荐）
   - 乐观锁（version字段）
```

### 3. redo log、undo log、binlog各自干什么？
```
核心：按事务提交的时间线理解

1. Undo Log：
   - 支持回滚和MVCC
   - 记录相反的操作

2. Redo Log：
   - 保证持久性
   - 记录物理修改

3. Binlog：
   - 主从复制、归档
   - 记录逻辑变更

4. 两阶段提交：
   - 保证Redo Log和Binlog一致
```

```
核心：看访问路径，不是看有没有索引

1. 索引失效：
   - 字段上有函数
   - 隐式类型转换
   - LIKE '%abc'

2. 索引选择错误：
   - MySQL优化器选错了索引

3. 回表次数多：
   - 二级索引需要回表

4. 深分页：
   - LIMIT 1000000, 20

5. 数据量太大：
   - 即使用了索引，扫描行数还是很多
```

## 💡 面试回答技巧

### 1. 不要背定义，讲原理
```
❌ 错误回答：
"B+树是一种多路平衡查找树..."

✅ 正确回答：
"InnoDB用B+树主要是为了减少磁盘IO次数。
磁盘IO比内存慢10万倍，是数据库的主要瓶颈。
红黑树100万数据需要20层，20次IO。
B+树每个节点能存上千个key，只需要3层，3次IO。
而且B+树的叶子节点是有序链表，范围查询可以直接顺序扫描..."
```

### 2. 结合场景，讲实战
```
❌ 错误回答：
"事务有四个隔离级别..."

✅ 正确回答：
"我们项目遇到过库存超卖的问题。
一开始只加了@Transactional，但还是会超卖。
后来发现是因为普通SELECT是快照读，不加锁。
两个事务同时查询库存，都读到库存充足，都执行扣减，就超卖了。
最后改成直接UPDATE带库存检查，一条SQL解决了问题..."
```

### 3. 从底层往上讲
```
❌ 错误回答：
"联合索引要满足最左前缀原则..."

✅ 正确回答：
"联合索引(a,b,c)在B+树中是按(a,b,c)整体排序的。
先按a排序，a相同再按b排序，b相同再按c排序。
所以a是全局有序的，可以快速定位。
但如果跳过a直接查b，b在全局范围内是无序的，就用不上索引了。
这就是最左前缀原则的底层原因..."
```

## 🎯 总结

```
MySQL面试的本质：
不是背名词，是理解底层原理和执行过程

四条主线：
1. 数据怎么找 → B+树、索引、最左匹配
2. 并发下怎么不错 → MVCC、锁、快照读、当前读
3. 宕机后怎么活 → redo log、undo log、binlog
4. 为什么会慢 → EXPLAIN、访问路径、优化技巧

核心技巧：
1. 从底层往上讲（B+树结构 → 最左匹配）
2. 结合场景讲实战（库存超卖 → 快照读不加锁）
3. 按时间线讲流程（事务提交 → redo/undo/binlog）
4. 用排查顺序讲优化（EXPLAIN → type → key → rows）

记住：
面试官想看的不是你背了多少名词，
而是你有没有真正理解MySQL的底层原理，
有没有真正被线上问题打过脸。
```
