## Redis 三种高可用架构全景对比

> **开篇导读**：Redis 提供了三种高可用架构方案，每种方案解决不同的问题。在深入学习之前，先建立全局视角，了解它们的核心差异和适用场景。

### 🏗️ 三种架构概览

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    Redis 高可用架构演进路线                               │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  单机 Redis  →  主从复制  →  哨兵模式  →  Cluster 集群                   │
│     ↓              ↓            ↓              ↓                        │
│  性能最高      读写分离    自动故障转移    数据分片 + 高可用               │
│  无高可用      手动切换    自动切换        自动切换 + 水平扩展             │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘
```

### 📊 架构1：主从复制（Master-Slave Replication）

```
架构图：

         ┌─────────┐
         │ Master  │ ← 所有写操作
         │ (主库)  │
         └────┬────┘
              │ 异步复制
      ┌───────┼───────┐
      │       │       │
  ┌───▼───┐ ┌─▼─────┐ ┌─▼─────┐
  │Slave1 │ │Slave2 │ │Slave3 │ ← 读操作
  │(从库) │ │(从库) │ │(从库) │
  └───────┘ └───────┘ └───────┘

核心特点：
✅ 读写分离：Master 写，Slave 读
✅ 数据冗余：Slave 是 Master 的完整副本
✅ 负载均衡：多个 Slave 分担读请求

核心问题：
❌ 手动故障转移：Master 宕机需要人工介入
❌ 写性能瓶颈：所有写操作都在 Master
❌ 容量瓶颈：每个节点存储全量数据

适用场景：
- 读多写少（读写比 > 10:1）
- 数据量不大（< 100GB）
- 可接受短暂的手动切换
- 对成本敏感
```

### 📊 架构2：哨兵模式（Sentinel）

```
架构图：

    ┌──────────┐  ┌──────────┐  ┌──────────┐
    │Sentinel1 │  │Sentinel2 │  │Sentinel3 │ ← 监控 + 自动故障转移
    └─────┬────┘  └─────┬────┘  └─────┬────┘
          │             │             │
          └─────────────┼─────────────┘
                        │
        ┌───────────────┴───────────────┐
        │                               │
    ┌───▼───┐                     ┌─────▼──┐
    │ Master│ ← 写                │ Slave  │ ← 读
    └───────┘                     └────────┘

核心特点：
✅ 自动故障转移：Master 宕机自动选举新 Master
✅ 监控告警：实时监控 Redis 节点健康状态
✅ 配置中心：客户端通过 Sentinel 获取 Master 地址
✅ 高可用：无需人工介入，秒级切换

核心问题：
❌ 写性能瓶颈：仍然是单 Master 写入
❌ 容量瓶颈：无法水平扩展
❌ 需要额外部署：至少 3 个 Sentinel 节点

适用场景：
- 读多写少
- 数据量不大（< 100GB）
- 高可用要求高（99.9%+）
- 中小型应用（QPS < 10万）
```

### 📊 架构3：Cluster 集群

```
架构图：

    ┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐
    │   Master 1      │  │   Master 2      │  │   Master 3      │
    │ Slot: 0-5460    │  │ Slot: 5461-     │  │ Slot: 10923-    │
    │ (5461个槽)      │  │ 10922 (5462个)  │  │ 16383 (5461个)  │
    └────────┬────────┘  └────────┬────────┘  └────────┬────────┘
             │                    │                    │
             │ 复制               │ 复制               │ 复制
             │                    │                    │
    ┌────────▼────────┐  ┌────────▼────────┐  ┌────────▼────────┐
    │   Slave 1       │  │   Slave 2       │  │   Slave 3       │
    │ 备份 Master1    │  │ 备份 Master2    │  │ 备份 Master3    │
    └─────────────────┘  └─────────────────┘  └─────────────────┘

核心特点：
✅ 数据分片：16384 个 slot，数据分散存储
✅ 水平扩展：增加 Master 可线性提升性能和容量
✅ 自动故障转移：Master 宕机自动选举
✅ 无中心化：所有节点地位平等
✅ 客户端智能路由：自动路由到对应节点

核心问题：
❌ 复杂度高：需要理解 slot、路由、重定向
❌ 跨节点操作受限：不支持跨 slot 的事务、Lua 脚本
❌ 运维成本高：节点多，监控和管理复杂

适用场景：
- 读写都很高（QPS > 10万）
- 数据量大（> 100GB）
- 需要水平扩展
- 大型应用（亿级用户）
```

### 🔄 主从复制机制说明

> **重要**：无论是主从复制、哨兵模式，还是 Cluster 集群，底层的**主从复制机制都是完全相同的**：
> - PSYNC 协议（全量复制 + 部分复制）
> - 复制积压缓冲区（Replication Backlog）
> - 异步复制机制
> - 心跳检测（REPLCONF ACK）
>
> 区别在于：
> - **主从复制**：只有复制功能，手动故障转移
> - **哨兵模式**：主从复制 + 自动故障转移
> - **Cluster 集群**：主从复制 + 数据分片 + 自动故障转移

## 一、持久化机制：Redis 的记忆系统

### 🤔 为什么需要持久化？

想象一下，Redis 就像一个超级大脑，所有数据都存储在"内存"中，运行速度飞快。但是有一个致命问题：**一旦断电，所有记忆都会消失**！

```
没有持久化的 Redis：
┌─────────────────────────────────────────────────────────────┐
│ Redis 服务器（内存）                                           │
│ 用户数据：1000万条                                            │
│ 订单数据：500万条                                             │
│ 商品缓存：200万条                                             │
│                                                              │
│ 突然断电... ⚡                                                │
│ 重启后：所有数据丢失！❌                                       │
└─────────────────────────────────────────────────────────────┘

有持久化的 Redis：
┌─────────────────────────────────────────────────────────────┐
│ Redis 服务器（内存）                                           │
│ 用户数据：1000万条  ──┐                                       │
│ 订单数据：500万条   ──┼──> 定期保存到磁盘                      │
│ 商品缓存：200万条   ──┘                                       │
│                                                              │
│ 突然断电... ⚡                                                │
│ 重启后：从磁盘恢复数据 ✅                                      │
└─────────────────────────────────────────────────────────────┘
```

Redis 提供了两种持久化方式：
- **RDB（快照）**：定期拍照，记录某个时刻的完整状态
- **AOF（日志）**：记录每一个操作，像写日记一样
- **混合持久化**：结合 RDB 和 AOF 的优点（Redis 4.0+）

---

### 1.1 RDB：Redis 的"时光相册"

#### 💡 生活类比

RDB 就像你用相机给房间拍照：
- **拍照时刻**：记录当前房间的完整状态
- **照片文件**：dump.rdb 文件
- **恢复场景**：看着照片，可以快速还原房间布局

#### 🔧 RDB 配置详解

```bash
# redis.conf 配置

# ============ 自动触发规则 ============
save 900 1      # 15分钟内至少1个key变化 → 适合低频更新
save 300 10     # 5分钟内至少10个key变化 → 适合中频更新
save 60 10000   # 1分钟内至少10000个key变化 → 适合高频更新

# ============ RDB 文件配置 ============
dbfilename dump.rdb           # RDB 文件名
dir /var/lib/redis            # RDB 文件保存目录
rdbcompression yes            # 开启压缩（推荐）
rdbchecksum yes               # 开启 CRC64 校验（推荐）

# ============ 错误处理 ============
stop-writes-on-bgsave-error yes  # BGSAVE 失败时停止写入

# ============ 性能优化 ============
rdb-save-incremental-fsync yes   # 增量 fsync（Redis 7.0+）
```

#### ⚙️ RDB 触发机制：三种方式

```
方式1：自动触发（BGSAVE）
├─ 触发条件：满足 save 规则
├─ 执行方式：后台 fork 子进程
└─ 影响：几乎无影响（COW 机制）

方式2：手动触发（SAVE）
├─ 命令：redis-cli SAVE
├─ 执行方式：主进程直接生成 RDB
└─ 影响：阻塞所有客户端请求 ⚠️（生产禁用）

方式3：手动触发（BGSAVE）
├─ 命令：redis-cli BGSAVE
├─ 执行方式：后台 fork 子进程
└─ 影响：几乎无影响

方式4：自动触发（SHUTDOWN）
├─ 触发条件：执行 SHUTDOWN 命令
├─ 执行方式：主进程生成 RDB 后关闭
└─ 影响：阻塞关闭过程

方式5：自动触发（主从复制）
├─ 触发条件：Slave 首次连接 Master
├─ 执行方式：Master 执行 BGSAVE
└─ 影响：全量复制时自动触发
```

#### 🔬 BGSAVE 底层原理：深度剖析

**1. fork() 系统调用详解**

```
fork() 的魔法：

┌─────────────────────────────────────────────────────────────────┐
│ 执行前：Redis 主进程（10GB 内存）                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  内存布局：                                                       │
│  ┌────────────────────────────────────────────────────────┐    │
│  │ 数据区（10GB）：                                         │    │
│  │ ├─ String: user:1 = "Alice"                            │    │
│  │ ├─ Hash: order:100 = {id:100, price:99.9}              │    │
│  │ ├─ List: queue = [1, 2, 3, 4, 5]                       │    │
│  │ └─ ...                                                  │    │
│  └────────────────────────────────────────────────────────┘    │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘

执行 fork()：

┌─────────────────────────────────────────────────────────────────┐
│ 主进程（PID: 1000）           子进程（PID: 1001）                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌──────────────┐              ┌──────────────┐                │
│  │ 页表（指针） │              │ 页表（指针） │                │
│  └──────┬───────┘              └──────┬───────┘                │
│         │                             │                         │
│         └─────────────┬───────────────┘                         │
│                       ↓                                         │
│         ┌─────────────────────────────┐                        │
│         │  共享物理内存（10GB）        │                        │
│         │  所有数据页都标记为只读      │                        │
│         └─────────────────────────────┘                        │
│                                                                  │
│  特点：                                                          │
│  ✅ 不复制物理内存（10GB）                                       │
│  ✅ 只复制页表（几 MB）                                          │
│  ✅ 耗时：O(1)，与数据量无关（通常 < 100ms）                     │
└─────────────────────────────────────────────────────────────────┘

fork() 性能数据：
- 1GB 数据：fork 耗时 10-20ms
- 10GB 数据：fork 耗时 50-100ms
- 100GB 数据：fork 耗时 500ms-1s
```

**2. Copy-On-Write（COW）机制详解**

```
COW 工作流程：

场景：主进程修改 key "user:1"

┌─────────────────────────────────────────────────────────────────┐
│ 步骤1：主进程尝试写入                                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  主进程：SET user:1 "Bob"                                        │
│  ├─ 尝试修改内存页                                               │
│  └─ 触发写保护异常（Page Fault）                                 │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│ 步骤2：操作系统处理 Page Fault                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  操作系统：                                                       │
│  ├─ 检测到写保护页                                               │
│  ├─ 分配新的物理内存页（4KB）                                    │
│  ├─ 复制原始页内容到新页                                         │
│  └─ 更新主进程页表，指向新页                                     │
│                                                                  │
│  ┌──────────────┐              ┌──────────────┐                │
│  │ 主进程页表   │              │ 子进程页表   │                │
│  └──────┬───────┘              └──────┬───────┘                │
│         │                             │                         │
│         ↓                             ↓                         │
│  ┌──────────────┐              ┌──────────────┐                │
│  │ 新内存页     │              │ 原始内存页   │                │
│  │ user:1="Bob" │              │ user:1="Alice"│               │
│  └──────────────┘              └──────────────┘                │
```

#### 🔬 COW 机制深度剖析

**COW 的底层实现细节**

```
1. 页表标志位设置

Linux 页表项（PTE）结构：
┌─────────────────────────────────────────────────────────────────┐
│ 64 位页表项（PTE）                                               │
├─────────────────────────────────────────────────────────────────┤
│ 位 [51:12]：物理页帧号（PFN）                                    │
│ 位 [11:0]：标志位                                                │
│   ├─ Present (P)：页是否在内存中                                │
│   ├─ Read/Write (R/W)：页是否可写                               │
│   ├─ User/Supervisor (U/S)：用户态/内核态                       │
│   ├─ Accessed (A)：页是否被访问过                               │
│   ├─ Dirty (D)：页是否被修改过                                  │
│   └─ COW：Copy-On-Write 标志（软件定义）                        │
│                                                                  │
│ COW 页的标志位设置：                                             │
│ ├─ Present = 1（页在内存中）                                    │
│ ├─ R/W = 0（只读，触发 Page Fault）                             │
│ ├─ COW = 1（标识这是 COW 页）                                   │
│ └─ 引用计数 > 1（多个进程共享）                                  │
└─────────────────────────────────────────────────────────────────┘

2. Page Fault Handler 流程

伪代码：
void do_page_fault(address, error_code) {
    pte = get_page_table_entry(address);
    
    // 检查是否是 COW 页
    if (pte.present && !pte.writable && pte.is_cow) {
        page = pte.physical_page;
        
        // 检查引用计数
        if (page.ref_count == 1) {
            // 只有一个进程引用，直接设置为可写
            pte.writable = true;
            flush_tlb(address);
            return;
        }
        
        // 多个进程引用，需要复制
        new_page = alloc_page();           // 分配新页（0.5μs）
        copy_page(new_page, page);         // 复制 4KB（0.5μs）
        pte.physical_page = new_page;      // 更新页表
        pte.writable = true;               // 设置可写
        flush_tlb(address);                // 刷新 TLB
        page.ref_count--;                  // 减少引用计数
    }
}

3. 性能开销详解

单次 COW 开销：
├─ Page Fault 处理：0.5μs
├─ 分配新页：0.5μs
├─ 复制 4KB 数据：0.5μs
├─ 更新页表：0.1μs
├─ 刷新 TLB：0.1μs
└─ 总计：约 1.7μs

对比普通内存写入：
├─ 普通写入：50ns
├─ COW 写入：1.7μs
└─ 性能下降：34 倍（仅首次写入）

4. 高写入场景的影响

场景：100MB/s 写入速度
├─ 每秒触发 COW：100MB / 4KB = 25,600 次
├─ 每秒 COW 开销：25,600 × 1.7μs = 43.5ms
├─ 性能影响：4.35%
└─ 额外内存：100MB/s（持续增长）

风险：
❌ 如果 BGSAVE 耗时 60 秒
❌ 额外内存：100MB/s × 60s = 6GB
❌ 可能导致 OOM
```

**4. 数据库系统：MVCC（多版本并发控制）**

```
MySQL InnoDB MVCC（COW 思想）：

原理：
├─ 每个事务看到的是数据的一个快照
├─ 不同事务可以同时读写同一行数据
└─ 使用 COW 思想实现版本链

┌─────────────────────────────────────────────────────────────────┐

└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐

│ ├─ roll_ptr → 指向旧版本（Undo Log）                            │
│ └─ 旧版本保留（用于其他事务）                                    │
└─────────────────────────────────────────────────────────────────┘

✅ 修改时创建新版本（Copy）
✅ 旧版本保留（用于其他事务）
✅ 最终清理旧版本（Purge 线程）

├─ 每个版本有 xmin（创建事务 ID）和 xmax（删除事务 ID）
└─ 根据事务快照决定可见性

│ Tuple 1: id=1, name="Alice", age=25, xmin=100, xmax=101         │

└─────────────────────────────────────────────────────────────────┘

可见性判断：
├─ 事务 T1（xid=101）：看到 Tuple 2
├─ 事务 T2（xid=99）：看到 Tuple 1
└─ 事务 T3（xid=102）：看到 Tuple 2

优势：
✅ 读不阻塞写，写不阻塞读
✅ 高并发性能
✅ 实现 ACID 隔离级别

劣势：
❌ 需要定期清理旧版本（VACUUM）
❌ 表膨胀问题
❌ 更新性能略低（需要创建新版本）
```

#### 💡 COW 设计原则与最佳实践

```
1. 延迟复制原则
   ├─ 不提前复制，只在需要时复制
   ├─ 适用于读多写少的场景
   └─ 示例：Redis BGSAVE、Docker 镜像

2. 共享优先原则
   ├─ 尽可能共享数据
   ├─ 减少内存/磁盘占用
   └─ 示例：fork() 共享父进程内存

3. 隔离保证原则
   ├─ 修改时必须隔离
   ├─ 保证数据一致性
   └─ 示例：MVCC 版本隔离

4. 性能权衡原则
   ├─ 读性能：无影响（共享）
   ├─ 写性能：首次写入有开销（复制）
   └─ 空间：按需增长

5. 引用计数原则
   ├─ 跟踪共享资源的引用数
   ├─ 引用数为 0 时释放资源
   └─ 示例：Linux 物理页引用计数

最佳实践：

1. 评估写入频率
   ├─ 写入频率低：COW 效果好
   └─ 写入频率高：考虑其他方案

2. 预留足够内存
   ├─ Redis：预留 50% 内存用于 COW
   └─ 避免 OOM

3. 监控 COW 开销
   ├─ 监控额外内存占用
   ├─ 监控 Page Fault 次数
   └─ 及时调整策略

4. 在低峰期执行
   ├─ BGSAVE 在低峰期执行
   └─ 减少 COW 开销
```

```
│ 3. 辅助字段（可选）                                              │
│    ├─ redis-ver：Redis 版本（如 6.2.6）                         │

│    ├─ ctime：创建时间戳                                          │
│    └─ used-mem：使用的内存大小                                   │
│                                                                  │
│ 4. 数据库选择器（1 字节）                                        │
│    ├─ 0xFE（标识数据库编号）                                     │
│    └─ 数据库编号（如 0）                                         │
│                                                                  │
│ 5. 数据库大小信息（可选）                                        │
│    ├─ 0xFB（标识大小信息）                                       │
│    ├─ key 数量                                                   │
│    └─ 带过期时间的 key 数量                                      │
│                                                                  │
│ 6. 键值对数据（核心部分）                                        │
│    ┌─────────────────────────────────────────────────────────┐ │
│    │ 键值对1：                                                 │ │
│    │ ├─ 过期时间（可选，8 字节）                               │ │
│    │ │  └─ 0xFD（秒级）或 0xFC（毫秒级）                       │ │
│    │ ├─ 类型标识（1 字节）                                     │ │
│    │ │  ├─ 0：String                                          │ │
│    │ │  ├─ 1：List                                            │ │
│    │ │  ├─ 2：Set                                             │ │
│    │ │  ├─ 3：Sorted Set                                      │ │
│    │ │  ├─ 4：Hash                                            │ │
│    │ │  └─ ...                                                │ │
│    │ ├─ Key（字符串）                                          │ │
│    │ │  ├─ 长度（变长编码）                                   │ │
│    │ │  └─ 内容                                               │ │
│    │ └─ Value（根据类型编码）                                  │ │
│    │    ├─ String：长度 + 内容                                │ │
│    │    ├─ List：元素数量 + 每个元素                          │ │
│    │    ├─ Hash：字段数量 + 每个字段和值                      │ │
│    │    └─ ...                                                │ │
│    └─────────────────────────────────────────────────────────┘ │
│                                                                  │
│ 7. 数据库结束标识（1 字节）                                      │
│    └─ 0xFF（标识 RDB 文件结束）                                  │
│                                                                  │
│ 8. CRC64 校验和（8 字节）                                        │
│    ├─ 对整个 RDB 文件计算 CRC64                                  │
│    └─ 作用：检测文件损坏                                         │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘

示例：String 类型编码

原始数据：
SET user:1 "Alice"

RDB 编码：
┌────────────────────────────────────────┐
│ 0x00                # 类型：String      │
│ 0x06 "user:1"       # Key：长度 + 内容  │
│ 0x05 "Alice"        # Value：长度 + 内容│
└────────────────────────────────────────┘

压缩优化：
- LZF 压缩算法（rdbcompression yes）
- 整数编码优化（如 "123" → 整数 123）
- 字符串共享（相同字符串只存储一次）
```

#### 🚀 RDB 性能优化

**优化1：减少 fork() 耗时**

```
问题：fork() 耗时与内存大小成正比

原因：
- fork() 需要复制页表
- 页表大小 = 内存大小 / 页大小（4KB）
- 10GB 内存 → 2.5M 页表项 → 20MB 页表

解决方案：

1. 使用大页内存（Huge Pages）
   ├─ 页大小：4KB → 2MB
   ├─ 页表项：2.5M → 5K
   ├─ fork() 耗时：100ms → 10ms
   └─ 配置：
       echo 1024 > /proc/sys/vm/nr_hugepages
       echo always > /sys/kernel/mm/transparent_hugepage/enabled

2. 控制 Redis 内存大小
   ├─ 单实例不超过 20GB
   └─ 超过 20GB 使用 Cluster 分片

3. 避免高峰期执行 BGSAVE
   ├─ 使用 cron 定时任务
   └─ 在低峰期（凌晨）执行
```

**优化2：减少 COW 内存开销**

```
问题：写入密集时，COW 导致内存翻倍

场景：
- Redis 内存：10GB
- 写入速度：100MB/s
- BGSAVE 耗时：60 秒
- COW 内存：100MB/s × 60s = 6GB
- 总内存：10GB + 6GB = 16GB（可能 OOM）

解决方案：

1. 降低写入速度
   ├─ 业务层限流
   └─ 使用队列缓冲

2. 增加物理内存
   ├─ 预留 50% 内存给 COW
   └─ 10GB Redis → 15GB 物理内存

3. 使用 AOF 代替 RDB
   └─ AOF 重写也有 COW，但可控制频率
```

**优化3：减少磁盘 I/O**

```
问题：RDB 文件写入占用磁盘带宽

解决方案：

1. 使用 SSD
   ├─ 写入速度：100-500 MB/s
   └─ HDD：50-100 MB/s

2. 开启压缩（默认开启）
   rdbcompression yes
   ├─ 10GB 内存 → 5GB RDB 文件
   └─ 节省 50% 磁盘 I/O

3. 增量 fsync（Redis 7.0+）
   rdb-save-incremental-fsync yes
   ├─ 每写入 32MB 调用一次 fsync
   └─ 避免最后一次性 fsync 阻塞
```

#### ✅ RDB 优缺点（深度分析）

| 优点 | 缺点 |
|------|------|
| ✅ 文件紧凑，适合备份 | ❌ 数据丢失风险（最后一次快照后的数据） |
| ✅ 恢复速度快（10GB约1-2分钟） | ❌ fork 子进程耗时（数据量大时阻塞 100ms） |
| ✅ 性能影响小（主进程几乎无影响） | ❌ COW 内存开销（可能导致 OOM） |
| ✅ 适合灾难恢复（文件可异地备份） | ❌ 不适合实时持久化（分钟级） |
| ✅ 单文件，易于管理 | ❌ 大数据量时生成耗时长（分钟级） |

### 1.2 AOF：Redis 的"操作日志"

#### 💡 生活类比

AOF 就像你的银行流水账单：
- **每一笔交易**：都会记录下来
- **账单文件**：appendonly.aof 文件
- **恢复场景**：按照账单重放所有交易，就能还原账户状态

#### 🔧 AOF 配置详解

```bash
# redis.conf 配置

# ============ 开启 AOF ============
appendonly yes
appendfilename "appendonly.aof"
dir /var/lib/redis            # AOF 文件保存目录

# ============ 同步策略（核心配置）============
# 策略1：always（最安全，最慢）
appendfsync always    # 每个命令都同步，性能降低50%

# 策略2：everysec（推荐，平衡）⭐⭐⭐⭐⭐
appendfsync everysec  # 每秒同步，性能降低5%，最多丢失1秒数据

# 策略3：no（最快，最不安全）
appendfsync no        # 由操作系统决定，最多丢失30秒数据

# ============ AOF 重写配置 ============
auto-aof-rewrite-percentage 100   # 文件大小增长100%时重写
auto-aof-rewrite-min-size 64mb    # 文件至少64MB才重写
aof-rewrite-incremental-fsync yes # 增量 fsync（推荐）

# ============ AOF 加载配置 ============
aof-load-truncated yes            # 加载截断的 AOF 文件
```

#### 🔬 AOF 底层原理：深度剖析

**1. AOF 写入流程详解**

```

│                                                                  │
│  磁盘文件（appendonly.aof）：                                    │
│  ┌──────────────────────────────────────────────────────────┐  │
```

**3. AOF 重写原理详解**

```
问题：AOF 文件膨胀

场景：
SET counter 1
INCR counter    # counter = 2
INCR counter    # counter = 3
...
INCR counter    # counter = 10000

AOF 文件大小：10000 条命令 × 50 字节 = 500KB
实际需要：1 条命令（SET counter 10000）× 50 字节 = 50 字节
膨胀率：10 倍

解决方案：AOF 重写

┌─────────────────────────────────────────────────────────────────┐
│ AOF 重写流程（类似 BGSAVE）                                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│ 1. 触发条件                                                      │
│    ├─ 手动触发：BGREWRITEAOF 命令                                │
│    └─ 自动触发：                                                 │
│        ├─ 当前 AOF 文件大小 > auto-aof-rewrite-min-size         │
│        └─ 当前大小 / 上次重写后大小 > auto-aof-rewrite-percentage│
│                                                                  │
│ 2. fork() 子进程                                                 │
│    ├─ 耗时：50-100ms（10GB 数据）                               │
│    ├─ 主进程：继续处理请求                                       │
│    └─ 子进程：开始重写 AOF                                       │
│                                                                  │
│ 3. 子进程：遍历内存数据库                                        │
│    ├─ 不读取旧 AOF 文件                                          │
│    ├─ 直接从内存生成新 AOF                                       │
│    └─ 对每个 key：                                               │
│        ├─ String：SET key value                                 │
│        ├─ List：RPUSH key v1 v2 v3                              │
│        ├─ Hash：HMSET key f1 v1 f2 v2                           │
│        ├─ Set：SADD key m1 m2 m3                                │
│        └─ ZSet：ZADD key s1 m1 s2 m2                            │
│                                                                  │
│ 4. 主进程：双写机制                                              │
│    ├─ 写入旧 AOF 文件                                            │
│    └─ 写入 AOF 重写缓冲区（aof_rewrite_buf）                     │
│        └─ 用于同步重写期间的新命令                               │
│                                                                  │
│ 5. 子进程完成，通知主进程                                        │
│    └─ 通过信号（SIGCHLD）                                        │
│                                                                  │
│ 6. 主进程：追加增量数据                                          │
│    ├─ 阻塞主进程（重要）⚠️                                       │
│    ├─ 将 AOF 重写缓冲区追加到新 AOF 文件                         │
│    ├─ 耗时：取决于缓冲区大小（通常 < 1 秒）                      │
│    └─ 原子替换：rename(temp.aof, appendonly.aof)                │
│                                                                  │
│ 7. 删除旧 AOF 文件                                               │
│    └─ 重写完成 ✅                                                │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘

性能数据（10GB 数据）：
- fork()：50-100ms
- 重写 AOF：30-60 秒
- 追加增量：0.5-1 秒（阻塞）⚠️
- 总耗时：约 30-60 秒

文件大小对比：
- 旧 AOF：1GB
- 新 AOF：200MB（减少 80%）
```

#### 🚀 AOF 性能优化

**优化1：减少 AOF 文件大小**

```
问题：AOF 文件过大，加载慢

解决方案：

1. 定期执行 AOF 重写
   ├─ 手动：redis-cli BGREWRITEAOF
   └─ 自动：
       auto-aof-rewrite-percentage 100
       auto-aof-rewrite-min-size 64mb

2. 调整重写阈值
   ├─ 小系统：64MB
   ├─ 中型系统：256MB
   └─ 大型系统：1GB

3. 使用混合持久化（Redis 4.0+）
   aof-use-rdb-preamble yes
   ├─ AOF 文件 = RDB 格式 + AOF 增量
   └─ 文件大小减少 50%，加载速度提升 5 倍
```

**优化2：减少 fsync 阻塞**

```
问题：fsync 阻塞主线程

场景：
- appendfsync always
- 每个命令都 fsync
- 磁盘慢（HDD）：10-50ms
- 导致 QPS 降低 50%

解决方案：

1. 使用 everysec 策略（推荐）
   appendfsync everysec
   ├─ 后台线程 fsync
   └─ 不阻塞主线程

2. 使用 SSD
   ├─ fsync 耗时：1-5ms
   └─ HDD：10-50ms

3. 使用 no-appendfsync-on-rewrite
   no-appendfsync-on-rewrite yes
   ├─ AOF 重写期间不 fsync
   └─ 避免磁盘 I/O 竞争
```

**优化3：减少重写阻塞**

```
问题：AOF 重写最后阶段阻塞主线程

原因：
- 主进程追加 AOF 重写缓冲区
- 缓冲区大小：取决于重写期间的写入量
- 阻塞时间：0.5-2 秒

解决方案：

1. 控制写入速度
   ├─ 业务层限流
   └─ 避免重写期间大量写入

2. 在低峰期重写
   ├─ 使用 cron 定时任务
   └─ 凌晨执行 BGREWRITEAOF

3. 增量 fsync（Redis 7.0+）
   aof-rewrite-incremental-fsync yes
   ├─ 每写入 32MB 调用一次 fsync
   └─ 减少最后阶段的阻塞时间
```

### 1.3 混合持久化：鱼和熊掌兼得（Redis 4.0+）

#### 💡 核心思想

混合持久化结合了 RDB 和 AOF 的优点：
- **RDB 的快速恢复**
- **AOF 的数据安全**

```bash
# 开启混合持久化
aof-use-rdb-preamble yes    # 推荐 ⭐⭐⭐⭐⭐

# 工作原理：
# 1. AOF 重写时：将内存数据以 RDB 格式写入 AOF 文件开头
# 2. 后续命令：以 AOF 格式追加
# 3. 恢复时：先加载 RDB 部分（快），再重放 AOF 部分（增量）

# 优势：
# ✅ 恢复速度：接近 RDB（2-3分钟）
# ✅ 数据安全：接近 AOF（最多丢失1秒）
# ✅ 文件大小：比纯 AOF 小50%
```

### 1.4 持久化命令冲突与互斥机制

> **为什么需要了解命令冲突？**
> 
> 在生产环境中，BGSAVE 和 BGREWRITEAOF 都会 fork 子进程，消耗大量系统资源。Redis 设计了互斥机制来避免多个持久化操作同时进行，导致系统资源耗尽。理解这些机制对于合理安排持久化操作至关重要。

#### 🔒 命令互斥规则详解

**规则总览：**

```
┌─────────────────────────────────────────────────────────────────┐
│ Redis 持久化命令互斥矩阵                                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│ 当前正在执行的操作 │ 新命令           │ 处理方式              │
│ ──────────────────┼─────────────────┼───────────────────────│
│ BGSAVE            │ SAVE             │ ❌ 拒绝执行           │
│ BGSAVE            │ BGSAVE           │ ❌ 拒绝执行           │
│ BGSAVE            │ BGREWRITEAOF     │ ⏳ 延迟执行（排队）   │
│ ──────────────────┼─────────────────┼───────────────────────│
│ BGREWRITEAOF      │ SAVE             │ ❌ 拒绝执行           │
│ BGREWRITEAOF      │ BGSAVE           │ ❌ 拒绝执行           │
│ BGREWRITEAOF      │ BGREWRITEAOF     │ ❌ 拒绝执行           │
│ ──────────────────┼─────────────────┼───────────────────────│
│ SAVE              │ 任何命令         │ ⏳ 阻塞等待           │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘

核心原则：
1. BGSAVE 和 BGREWRITEAOF 不能同时执行（避免双重 fork）
2. SAVE 会阻塞所有命令（生产环境禁用）
3. BGREWRITEAOF 优先级高于 BGSAVE（数据安全优先）
```

#### 📋 场景详解

**场景1：BGSAVE 期间收到新的持久化命令**

```
┌─────────────────────────────────────────────────────────────────┐

│          "ERR Background save already in progress"               │
│                                                                  │

│ ├─ SAVE 是同步操作，会阻塞主进程                                 │

│                                                                  │

┌─────────────────────────────────────────────────────────────────┐

├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│ 时间线：                                                          │
│ T0: 执行 BGSAVE（自动触发或手动）                                │
│     └─ 子进程 PID: 12345，正在生成 RDB                          │
│                                                                  │

│      └─ Redis 返回错误：                                         │
│          "ERR Background save already in progress"               │
│                                                                  │
│ 原因：                                                            │
│ ├─ 避免多次 fork() 消耗大量内存                                  │
│ ├─ 避免生成多个 RDB 文件造成混乱                                 │
│ └─ 第一个 BGSAVE 完成后，数据已经是最新的                        │
│                                                                  │
│ 处理方式：                                                        │
│ ✅ 拒绝执行第二个 BGSAVE                                         │
│ ✅ 返回错误信息                                                  │
│ ✅ 客户端可以等待第一个 BGSAVE 完成                              │
│                                                                  │
│ 验证方法：                                                        │
│ redis-cli INFO persistence                                       │

│                                                                  │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│ 场景1.3：BGSAVE 期间收到 BGREWRITEAOF 命令 ⭐重要                │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│ 时间线：                                                          │
│ T0: 执行 BGSAVE                                                  │
│     └─ 子进程正在生成 RDB 文件（预计 60 秒）                     │
│                                                                  │
│ T10: 客户端发送 BGREWRITEAOF 命令                                │
│      └─ Redis 返回：                                             │
│          "Background append only file rewriting scheduled"       │
│                                                                  │
│ T60: BGSAVE 完成                                                 │
│      └─ Redis 自动触发 BGREWRITEAOF                              │
│          ├─ fork() 新的子进程                                    │
│          └─ 开始 AOF 重写                                        │
│                                                                  │
│ 原因：                                                            │
│ ├─ AOF 重写优先级高于 RDB（数据安全性更重要）                    │
│ ├─ 但不能同时执行（避免双重 fork）                               │
│ └─ 因此采用延迟执行策略                                          │

│ if (server.rdb_child_pid != -1) {  // BGSAVE 正在执行           │
│     server.aof_rewrite_scheduled = 1;  // 设置延迟标志          │
│     return C_OK;  // 返回成功                                    │
│ }                                                                │
│                                                                  │
│ // BGSAVE 完成后的回调                                           │
│ if (server.aof_rewrite_scheduled) {                              │
│     rewriteAppendOnlyFileBackground();  // 执行 AOF 重写        │
│ }                                                                │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```


#### ⚠️ 生产环境最佳实践

**1. 避免持久化操作冲突**

```
问题：业务高峰期频繁触发 BGSAVE，导致 BGREWRITEAOF 被延迟

场景：
├─ 配置：save 60 10000（60 秒内有 1 万次写入就触发 BGSAVE）
├─ 业务高峰：每分钟 5 万次写入
├─ 结果：每 60 秒触发一次 BGSAVE
└─ 问题：BGREWRITEAOF 一直被延迟，AOF 文件越来越大

解决方案：

方案1：调整 BGSAVE 触发条件（推荐）⭐⭐⭐⭐⭐
save 900 1        # 15 分钟内至少 1 次写入
save 300 10       # 5 分钟内至少 10 次写入
save 60 10000     # 1 分钟内至少 1 万次写入

优势：
✅ 降低 BGSAVE 触发频率
✅ 给 BGREWRITEAOF 留出执行时间
✅ 减少 fork() 次数

方案2：在低峰期执行持久化操作（推荐）⭐⭐⭐⭐⭐
# 使用 cron 定时任务
0 2 * * * redis-cli BGREWRITEAOF  # 凌晨 2 点执行 AOF 重写
0 3 * * * redis-cli BGSAVE        # 凌晨 3 点执行 RDB 备份

优势：
✅ 避开业务高峰期
✅ 减少对业务的影响
✅ 持久化操作不会冲突

方案3：使用混合持久化（推荐）⭐⭐⭐⭐⭐
aof-use-rdb-preamble yes

优势：
✅ AOF 重写时生成 RDB 格式，速度快
✅ 减少 AOF 文件大小
✅ 降低重写频率

方案4：监控持久化状态
redis-cli INFO persistence

关键指标：
rdb_bgsave_in_progress:0        # 0 表示没有 BGSAVE 在执行
aof_rewrite_in_progress:0       # 0 表示没有 AOF 重写在执行
aof_rewrite_scheduled:0         # 0 表示没有延迟的 AOF 重写
rdb_last_bgsave_time_sec:45     # 上次 BGSAVE 耗时 45 秒
aof_last_rewrite_time_sec:120   # 上次 AOF 重写耗时 120 秒

告警规则：
⚠️ 如果 aof_rewrite_scheduled = 1 持续超过 10 分钟
   → 说明 BGSAVE 频繁执行，阻塞了 AOF 重写
   → 需要调整 save 规则或业务限流
```

**2. 资源消耗评估**

```
场景：10GB Redis 实例，业务高峰期

假设：
- 写入速度：100MB/s
- BGSAVE 耗时：60 秒
- BGREWRITEAOF 耗时：120 秒

资源消耗分析：

1. 单独执行 BGSAVE：
   ├─ fork() 耗时：50-100ms
   ├─ COW 内存：100MB/s × 60s = 6GB
   ├─ 总内存：10GB + 6GB = 16GB
   └─ CPU：30%

2. 单独执行 BGREWRITEAOF：
   ├─ fork() 耗时：50-100ms
   ├─ COW 内存：100MB/s × 120s = 12GB
   ├─ 总内存：10GB + 12GB = 22GB
   └─ CPU：40%

3. 如果同时执行（假设没有互斥）：
   ├─ 两次 fork()：100-200ms
   ├─ COW 内存：6GB + 12GB = 18GB
   ├─ 总内存：10GB + 18GB = 28GB ⚠️ 可能 OOM
   └─ CPU：70%

结论：
✅ 互斥机制避免了同时执行，防止 OOM
✅ 但延迟执行可能导致 AOF 文件过大
✅ 需要合理安排持久化时间
```

**3. 常见问题排查**

```
问题1：BGREWRITEAOF 一直被延迟

症状：
redis-cli INFO persistence
aof_rewrite_scheduled:1  # 一直是 1
aof_current_size:5GB     # AOF 文件越来越大

原因：
BGSAVE 频繁执行，阻塞了 BGREWRITEAOF

排查步骤：
1. 检查 save 规则
   redis-cli CONFIG GET save
   
2. 检查写入频率
   redis-cli INFO stats
   instantaneous_ops_per_sec:50000  # 每秒 5 万次操作

3. 检查 BGSAVE 频率
   redis-cli INFO persistence
   rdb_changes_since_last_save:15000  # 距离上次保存已有 1.5 万次写入

解决方案：
1. 调整 save 规则（降低触发频率）
2. 手动执行 BGREWRITEAOF（低峰期）
3. 关闭 RDB，只使用 AOF

问题2：持久化操作耗时过长

症状：
rdb_last_bgsave_time_sec:300  # BGSAVE 耗时 5 分钟
aof_last_rewrite_time_sec:600 # AOF 重写耗时 10 分钟

原因：
1. 数据量过大（> 20GB）
2. 磁盘性能差（HDD）
3. 写入速度过快（COW 内存大）

解决方案：
1. 使用 SSD 磁盘
2. 控制单实例内存（< 20GB）
3. 使用 Cluster 分片
4. 降低写入速度（业务限流）

问题3：fork() 失败

症状：
redis-cli INFO persistence
rdb_last_bgsave_status:err
latest_fork_usec:0  # fork 失败

原因：
1. 内存不足（无法分配页表）
2. 系统限制（ulimit）

解决方案：
1. 增加物理内存
2. 调整系统参数：
   sysctl vm.overcommit_memory=1
3. 减少 Redis 内存使用
```

#### 📊 命令优先级总结

```
优先级排序（从高到低）：

1. SAVE（最高优先级，但阻塞所有操作）
   └─ 生产环境禁用 ⚠️

2. BGREWRITEAOF（高优先级）
   ├─ 数据安全性优先
   ├─ 可以延迟 BGSAVE
   └─ 不能被延迟

3. BGSAVE（低优先级）
   ├─ 可以被 BGREWRITEAOF 延迟
   └─ 不能同时执行多个

执行策略：
✅ 优先保证 AOF 重写（数据安全）
✅ 避免同时 fork 多个子进程（资源限制）
✅ 在低峰期执行持久化操作（性能优化）
```

## 二、主从复制：Redis 的分身术

> **重要说明**：Redis Cluster 中的主从复制机制与本章节讲解的**完全相同**。无论是普通主从、哨兵模式，还是 Cluster 集群，底层的主从复制原理、PSYNC 协议、复制积压缓冲区等机制都是一致的。Cluster 只是在此基础上增加了数据分片和自动故障转移功能。

### 🤔 为什么需要主从复制？

想象一下，你开了一家餐厅（Redis 单机）：
- **问题1**：只有一个厨师，客人太多忙不过来（性能瓶颈）
- **问题2**：厨师生病了，餐厅就关门了（单点故障）
- **解决方案**：招聘多个厨师，一个主厨（Master）+ 多个副厨（Slave）

```
主从复制架构：

        ┌─────────┐
        │  Master │ (主厨：处理所有写请求)
        └─────────┘
             │
      ┌──────┴──────┐
      │             │
  ┌───▼───┐   ┌───▼───┐
  │ Slave1│   │ Slave2│ (副厨：处理读请求)
  └───────┘   └───────┘

优势：
✅ 读写分离：Master 写，Slave 读
✅ 负载均衡：多个 Slave 分担读请求
✅ 数据冗余：Slave 是 Master 的备份
✅ 高可用：Master 宕机，Slave 可以顶上
```

### 2.1 主从复制配置

```bash
# Slave 配置
replicaof 192.168.1.100 6379  # 指定 Master
masterauth password            # Master 密码
replica-read-only yes          # 只读模式（默认）

# Master 配置（可选）
min-replicas-to-write 1        # 至少 1 个 Slave 在线才能写入
min-replicas-max-lag 10        # Slave 延迟不超过 10 秒

# 复制相关配置
repl-diskless-sync no          # 是否使用无盘复制（默认 no）
repl-diskless-sync-delay 5     # 无盘复制延迟时间
repl-backlog-size 1mb          # 复制积压缓冲区大小（重要）⭐
repl-backlog-ttl 3600          # 缓冲区保留时间（秒）
repl-timeout 60                # 复制超时时间
repl-disable-tcp-nodelay no    # 是否禁用 TCP_NODELAY
```

### 2.2 主从复制核心原理：深度剖析

#### 📡 复制的三个阶段

```
┌─────────────────────────────────────────────────────────────────┐
│ 阶段1：建立连接（Handshake）                                      │
│ 阶段2：数据同步（Sync）                                          │
│ 阶段3：命令传播（Command Propagation）                           │
└─────────────────────────────────────────────────────────────────┘
```

#### 🔌 阶段1：建立连接（Handshake）

```
详细流程：

1. Slave 连接 Master
   ├─ Slave 执行：REPLICAOF 192.168.1.100 6379
   └─ 建立 TCP 连接

2. Slave 发送 PING
   ├─ 检查网络连接是否正常
   ├─ Master 响应：+PONG
   └─ 如果超时，断开重连

3. 身份验证（如果配置了密码）
   ├─ Slave 发送：AUTH <password>
   ├─ Master 验证密码
   └─ 验证失败则断开连接

4. Slave 发送端口信息
   ├─ Slave 发送：REPLCONF listening-port 6380
   └─ Master 记录 Slave 的端口信息

5. Slave 发送 IP 地址
   ├─ Slave 发送：REPLCONF ip-address 192.168.1.101
   └─ Master 记录 Slave 的 IP 地址

6. Slave 发送能力声明
   ├─ Slave 发送：REPLCONF capa eof capa psync2
   └─ 告知 Master 支持的功能（EOF、PSYNC2 等）

7. Slave 发送 PSYNC 命令（进入同步阶段）
   ├─ 首次连接：PSYNC ? -1
   └─ 断线重连：PSYNC <runid> <offset>
```

#### 🔄 阶段2：数据同步（Sync）- 核心机制

**2.2.1 PSYNC 协议：部分同步的关键**

```
PSYNC 命令格式：
PSYNC <runid> <offset>

参数说明：
- runid：Master 的运行 ID（40 位随机字符串）
- offset：复制偏移量（Slave 已复制的数据位置）

Master 响应：
1. +FULLRESYNC <runid> <offset>  → 全量复制
2. +CONTINUE                     → 部分复制（增量）
3. -ERR                          → 错误
```

**2.2.2 全量复制（Full Resynchronization）**

```
详细流程（10GB 数据为例）：

┌─────────────────────────────────────────────────────────────────┐
│ Master 节点                          Slave 节点                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│ 1. 收到 PSYNC ? -1                                              │
│    ├─ 判断：首次连接                                             │
│    └─ 响应：+FULLRESYNC <runid> <offset>                        │
│                                      ├─ 收到响应                 │
│                                      └─ 准备接收 RDB             │
│                                                                  │
│ 2. fork() 子进程                                                │
│    ├─ 主进程：继续处理客户端请求                                 │
│    ├─ 子进程：生成 RDB 文件                                      │
│    └─ 耗时：10GB 数据约 30-60 秒                                │
│                                                                  │
│ 3. 主进程：缓存写命令                                            │
│    ├─ 创建复制缓冲区（Client Output Buffer）                    │
│    ├─ 将所有写命令写入缓冲区                                     │
│    └─ 缓冲区大小：client-output-buffer-limit slave 256mb 64mb 60│
│                                                                  │
│ 4. 子进程：生成 RDB 完成                                         │
│    ├─ RDB 文件大小：约 5GB（压缩后）                            │
│    └─ 通知主进程                                                 │
│                                                                  │
│ 5. 主进程：发送 RDB 文件                                         │
│    ├─ 通过 socket 发送 RDB                                      │
│    ├─ 传输速度：100MB/s                                         │
│    └─ 传输时间：5GB / 100MB/s = 50 秒                           │
│                                      ├─ 接收 RDB 文件            │
│                                      ├─ 清空旧数据               │
│                                      └─ 加载 RDB（30-60 秒）     │
│                                                                  │
│ 6. 主进程：发送缓冲区命令                                        │
│    ├─ 发送期间积累的写命令                                       │
│    └─ 命令数量：可能有几万到几十万条                             │
│                                      ├─ 执行缓冲区命令            │
│                                      └─ 同步完成 ✅              │
│                                                                  │
│ 7. 进入命令传播阶段                                              │
│    └─ 实时同步写命令                                             │
│                                      └─ 实时执行写命令            │
└─────────────────────────────────────────────────────────────────┘

总耗时：约 2-3 分钟（10GB 数据）
```

**2.2.3 部分复制（Partial Resynchronization）- Redis 2.8+**

```
核心数据结构：

1. 复制偏移量（Replication Offset）
   ├─ Master offset：Master 已发送的字节数
   ├─ Slave offset：Slave 已接收的字节数
   └─ 正常情况：Master offset = Slave offset

2. 复制积压缓冲区（Replication Backlog Buffer）
   ├─ 环形缓冲区（Ring Buffer）
   ├─ 默认大小：1MB（repl-backlog-size）
   ├─ 存储内容：最近的写命令
   └─ 作用：支持部分复制

3. Master 运行 ID（Run ID）
   ├─ 40 位随机字符串
   ├─ Master 启动时生成
   └─ 用于判断是否是同一个 Master

┌─────────────────────────────────────────────────────────────────┐
│ 复制积压缓冲区（Ring Buffer）                                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  [SET user:1 Alice][SET user:2 Bob][DEL cache:1][INCR counter]  │
│   ↑                                                       ↑      │
│   offset: 1000                                    offset: 1500   │
│                                                                  │
│  如果 Slave 断线重连，offset = 1200：                            │
│  ├─ offset 1200 在缓冲区内 ✅                                    │
│  ├─ Master 发送 offset 1200-1500 的命令                         │
│  └─ 部分复制成功                                                 │
│                                                                  │
│  如果 Slave 断线重连，offset = 500：                             │
│  ├─ offset 500 不在缓冲区内 ❌                                   │
│  ├─ 缓冲区已被覆盖                                               │
│  └─ 降级为全量复制                                               │
└─────────────────────────────────────────────────────────────────┘
```

**部分复制流程：**

```
场景：Slave 短暂断线后重连

1. Slave 发送 PSYNC <runid> <offset>
   ├─ runid：之前记录的 Master ID
   └─ offset：断线前的复制偏移量（如 1200）

2. Master 判断
   ├─ runid 是否匹配？
   │  ├─ 不匹配 → 全量复制（Master 重启过）
   │  └─ 匹配 → 继续判断
   │
   └─ offset 是否在缓冲区内？
      ├─ 在缓冲区内 → 部分复制 ✅
      └─ 不在缓冲区内 → 全量复制 ❌

3. Master 响应：+CONTINUE
   └─ 发送 offset 1200-1500 之间的命令

4. Slave 执行增量命令
   └─ 同步完成 ✅（耗时：毫秒级）

优势：
✅ 避免全量复制的开销
✅ 网络抖动不会导致大量数据传输
✅ 恢复速度快（毫秒级）
```

**2.2.4 复制积压缓冲区大小计算**

```
公式：
repl-backlog-size = 平均写入速度 × 断线时间 × 2

示例：
- 平均写入速度：10MB/s
- 预期断线时间：60 秒
- 计算：10MB/s × 60s × 2 = 1200MB

配置：
repl-backlog-size 1200mb

说明：
- × 2 是安全系数
- 如果缓冲区太小，容易触发全量复制
- 如果缓冲区太大，浪费内存

推荐配置：
- 小型系统：16MB
- 中型系统：64MB
- 大型系统：256MB-1GB
```

#### 📨 阶段3：命令传播（Command Propagation）

```
实时同步流程：

1. Master 接收写命令
   ├─ 客户端：SET user:100 Alice
   └─ Master 执行命令

2. Master 传播命令
   ├─ 写入 AOF 文件（如果开启）
   ├─ 写入复制积压缓冲区
   └─ 发送给所有 Slave

3. Slave 接收并执行
   ├─ 执行：SET user:100 Alice
   ├─ 更新 offset
   └─ 数据同步完成

4. 心跳机制
   ├─ Slave 每秒发送：REPLCONF ACK <offset>
   ├─ 作用1：报告复制进度
   ├─ 作用2：检测网络连接
   └─ 作用3：辅助实现 min-replicas-to-write
```

### 2.3 无盘复制（Diskless Replication）- Redis 2.8.18+

```
传统复制（有盘）：
Master → 生成 RDB 文件 → 写入磁盘 → 读取文件 → 发送给 Slave
         ↑                ↑
      耗时 30s         耗时 10s（磁盘 I/O）

无盘复制：
Master → 生成 RDB 数据 → 直接通过 socket 发送给 Slave
         ↑
      耗时 30s（省略磁盘 I/O）

配置：
repl-diskless-sync yes              # 开启无盘复制
repl-diskless-sync-delay 5          # 延迟 5 秒（等待更多 Slave）

优势：
✅ 节省磁盘 I/O（SSD 磨损）
✅ 速度更快（省略磁盘读写）
✅ 适合磁盘慢但网络快的场景

劣势：
❌ 如果传输失败，需要重新生成 RDB
❌ 多个 Slave 同时请求时，只能串行处理

适用场景：
- 磁盘慢（机械硬盘）
- 网络快（万兆网卡）
- Slave 数量少（1-2 个）
```

### 2.4 主从复制的性能优化

#### 🚀 优化1：避免全量复制

```
问题：全量复制开销大
- Master：fork 子进程，CPU 和内存开销
- 网络：传输 GB 级数据
- Slave：清空数据，加载 RDB

解决方案：
1. 增大复制积压缓冲区
   repl-backlog-size 256mb  # 根据实际情况调整

2. 避免 Master 重启
   ├─ 使用持久化（RDB/AOF）
   └─ 重启后 runid 会变化，导致全量复制

3. 网络稳定性
   ├─ 使用专用网络
   └─ 避免跨机房复制（延迟高）

4. 监控复制状态
   redis-cli INFO replication
   ├─ master_repl_offset：Master 偏移量
   ├─ slave_repl_offset：Slave 偏移量
   └─ repl_backlog_size：缓冲区大小
```

#### 🚀 优化2：减少复制延迟

```
问题：Slave 数据落后 Master

原因：
1. 网络延迟
2. Slave 性能不足
3. Master 写入速度过快

解决方案：
1. 禁用 TCP_NODELAY
   repl-disable-tcp-nodelay no  # 默认 no（推荐）
   
   说明：
   - no：立即发送数据（低延迟）
   - yes：合并数据包（省带宽，但延迟高）

2. 监控复制延迟
   redis-cli -h slave_ip INFO replication | grep master_last_io_seconds_ago
   
   ├─ < 1 秒：正常
   ├─ 1-5 秒：轻微延迟
   └─ > 5 秒：严重延迟（需要排查）

3. 使用 WAIT 命令（Redis 3.0+）
   WAIT 2 1000  # 等待 2 个 Slave 同步，超时 1 秒
   
   应用场景：
   - 关键数据写入后，确保同步到 Slave
   - 防止主从切换时数据丢失
```

#### 🚀 优化3：复制缓冲区溢出

```
问题：复制缓冲区溢出导致 Slave 断开

配置：
client-output-buffer-limit slave 256mb 64mb 60

参数说明：
- 256mb：硬限制（超过立即断开）
- 64mb：软限制（超过 60 秒后断开）
- 60：软限制持续时间（秒）

场景：
1. Master 写入速度：100MB/s
2. Slave 网络慢：50MB/s
3. 缓冲区积压：100MB/s - 50MB/s = 50MB/s
4. 5 秒后缓冲区达到 250MB
5. 触发硬限制，Slave 断开 ❌

解决方案：
1. 增大缓冲区
   client-output-buffer-limit slave 512mb 128mb 60

2. 提升 Slave 性能
   ├─ 升级网络带宽
   └─ 优化 Slave 配置

3. 降低 Master 写入速度
   └─ 业务层限流
```

### 2.5 主从复制常见问题

#### ❌ 问题1：复制风暴

```
场景：Master 重启，所有 Slave 同时请求全量复制

问题：
- Master：同时 fork 多个子进程，内存和 CPU 爆炸
- 网络：同时传输多个 GB 级 RDB 文件，带宽打满
- 结果：Master 宕机，雪崩 ❌

解决方案：
1. 分批重启 Slave
   ├─ 先重启 1-2 个 Slave
   ├─ 等待同步完成
   └─ 再重启其他 Slave

2. 使用树状复制（Redis 不原生支持，需要手动配置）
   Master → Slave1 → Slave2
                  └─ Slave3
   
   优势：
   ✅ Master 只需复制给 Slave1
   ✅ Slave2/3 从 Slave1 复制
   ✅ 降低 Master 压力

3. 使用 Cluster 模式
   ├─ 每个 Master 只有 1-2 个 Slave
   └─ 分散复制压力
```

#### ❌ 问题2：数据不一致

```
场景：Slave 数据与 Master 不一致

原因：
1. 复制延迟（异步复制）
2. 网络分区
3. Slave 被误写入（未设置 replica-read-only）

解决方案：
1. 强制只读
   replica-read-only yes  # 必须开启

2. 使用 WAIT 命令
   SET key value
   WAIT 2 1000  # 确保同步到 2 个 Slave

3. 监控复制状态
   redis-cli INFO replication
   ├─ master_repl_offset：Master 偏移量
   └─ slave0:offset：Slave 偏移量
   
   如果差距过大（> 1000），说明延迟严重

4. 应用层处理
   ├─ 关键读操作：读 Master
   └─ 普通读操作：读 Slave
```

#### ❌ 问题3：过期键不一致

```
场景：Slave 的过期键未被删除

原因：
- Slave 不会主动删除过期键
- 必须等待 Master 发送 DEL 命令

示例：
1. Master：SET key value EX 10  # 10 秒过期
2. 10 秒后，Master 删除 key，发送 DEL key
3. 如果网络延迟，Slave 可能在 11 秒后才删除
4. 期间读取 Slave，可能读到过期数据 ❌

解决方案：
1. Redis 3.2+ 改进
   ├─ Slave 读取时检查过期时间
   └─ 如果已过期，返回空值（但不删除）

2. 应用层处理
   ├─ 关键数据：读 Master
   └─ 设置合理的过期时间（不要太短）
```

## 三、哨兵（Sentinel）：Redis 的自动驾驶

### 🤔 为什么需要哨兵？

主从复制有一个问题：**Master 宕机后，需要手动切换**。

哨兵就像汽车的自动驾驶系统：
- **监控**：实时检测 Master 和 Slave 的健康状态
- **通知**：故障时通知管理员
- **自动故障转移**：Master 宕机时自动选举新 Master
- **配置提供**：客户端通过 Sentinel 获取 Master 地址

```
哨兵架构：

    ┌──────────┐  ┌──────────┐  ┌──────────┐
    │Sentinel1 │  │Sentinel2 │  │Sentinel3 │
    └─────┬────┘  └─────┬────┘  └─────┬────┘
          │             │             │
          └─────────────┼─────────────┘
                        │
        ┌───────────────┴───────────────┐
        │                               │
    ┌───▼───┐                     ┌─────▼──┐
    │ Master│                     │ Slave  │
    └───────┘                     └────────┘
```

### 3.1 哨兵配置

```bash
# sentinel.conf

# 监控 Master
sentinel monitor mymaster 192.168.1.100 6379 2
# 2：至少 2 个 Sentinel 认为 Master 下线才进行故障转移

# 判断下线时间
sentinel down-after-milliseconds mymaster 30000  # 30秒

# 故障转移超时时间
sentinel failover-timeout mymaster 180000  # 3分钟

# 同时进行同步的 Slave 数量
sentinel parallel-syncs mymaster 1
```

### 3.2 故障转移流程

```
1. 主观下线（SDOWN）：
   - 单个 Sentinel 认为 Master 下线

2. 客观下线（ODOWN）：
   - 多数 Sentinel 认为 Master 下线

3. 选举 Leader Sentinel：
   - 使用 Raft 算法选举

4. 故障转移：
   - 从 Slave 中选择新 Master
   - 让其他 Slave 复制新 Master
   - 通知客户端新 Master 地址
```

## 四、Cluster 集群：Redis 的分布式王国

> **完整补充版**：基于生产实践，深入讲解 Cluster 的架构、数据分片、故障转移、脑裂问题等核心内容。

### 🤔 为什么需要 Cluster？

哨兵解决了高可用问题，但还有两个核心问题没有解决：
1. **单个 Master 的容量有限**（内存瓶颈）
2. **单个 Master 的性能有限**（QPS 瓶颈）

Redis Cluster 是 Redis 的**分布式实现**，具有以下核心特性：
- **无中心架构**：没有中心节点，所有节点地位平等
- **数据分片**：16384 个哈希槽（Hash Slot）
- **自动故障转移**：不需要 Sentinel，Master 宕机自动选举
- **水平扩展**：可动态添加/删除节点，官方推荐不超过 1000 个节点
- **高性能**：性能优于哨兵模式，配置简单

```
Cluster 架构（3主3从）：

    ┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐
    │   Master 1      │  │   Master 2      │  │   Master 3      │
    │ 192.168.1.101   │  │ 192.168.1.102   │  │ 192.168.1.103   │
    │ Slot: 0-5460    │  │ Slot: 5461-     │  │ Slot: 10923-    │
    │ (5461个槽)      │  │ 10922 (5462个)  │  │ 16383 (5461个)  │
    └────────┬────────┘  └────────┬────────┘  └────────┬────────┘
             │                    │                    │
             │ 复制               │ 复制               │ 复制
             │                    │                    │
    ┌────────▼────────┐  ┌────────▼────────┐  ┌────────▼────────┐
    │   Slave 1       │  │   Slave 2       │  │   Slave 3       │
    │ 192.168.1.104   │  │ 192.168.1.105   │  │ 192.168.1.106   │
    │ 备份 Master1    │  │ 备份 Master2    │  │ 备份 Master3    │
    └─────────────────┘  └─────────────────┘  └─────────────────┘

特点：
✅ 每个 Master 都有 Slave 备份
✅ 所有节点互相连接（Full Mesh）
✅ 节点之间通过 Gossip 协议通信
✅ 客户端可以连接任意节点
```

### 4.1 数据分片：哈希槽（Hash Slot）

#### 💡 核心问题

切片集群要解决两大核心问题：
1. **数据切片后，在多个实例之间如何分布？**
2. **客户端怎么确定想要访问的数据在哪个实例上？**

#### 🎯 解决方案：16384 个哈希槽

Redis Cluster 使用 **16384 个哈希槽**来分配数据。

**为什么是 16384 个槽？**
- 足够大：可以支持足够多的节点
- 足够小：槽位信息可以快速传播（心跳包 2KB）
- 2^14 = 16384：方便位运算

#### 📊 槽位定位算法

```java
/**
 * 槽位计算：CRC16(key) % 16384
 */
public static int calculateSlot(String key) {
    // 1. 检查是否有 Hash Tag
    int start = key.indexOf('{');
    int end = key.indexOf('}', start + 1);
    
    if (start != -1 && end != -1 && end > start + 1) {
        // 使用 Hash Tag 内的内容计算
        key = key.substring(start + 1, end);
    }
    
    // 2. CRC16 算法计算 16 bit 值
    int crc = CRC16.crc16(key.getBytes());
    
    // 3. 对 16384 取模（使用位运算优化）
    return crc & 0x3FFF;  // 0x3FFF = 16383
}

// Hash Tag 示例：这些 key 会分配到同一个 slot
String key1 = "user:{123}:profile";  // Slot 5970
String key2 = "user:{123}:orders";   // Slot 5970
String key3 = "user:{123}:cart";     // Slot 5970
// 优势：可以使用 MGET、事务、Lua 脚本
```

#### 🏷️ Hash Tag：让相关数据聚在一起

##### 💡 什么是 Hash Tag？

**Hash Tag** 是 Redis Cluster 中的一种特殊机制，通过在 key 中使用 `{}` 标记，可以**控制多个 key 分配到同一个哈希槽**。

```
Hash Tag 语法：
key = "prefix:{tag}:suffix"

计算规则：
- 如果 key 包含 {}，且 {} 之间有内容
- 则只对 {} 之间的内容计算 CRC16
- 否则对整个 key 计算 CRC16

示例：
user:{100}:profile  → CRC16("100") % 16384 = Slot 5970
user:{100}:orders   → CRC16("100") % 16384 = Slot 5970
user:{100}:cart     → CRC16("100") % 16384 = Slot 5970
user:100:profile    → CRC16("user:100:profile") % 16384 = Slot 12345
```

##### 🎯 Hash Tag 有什么用？

在 Redis Cluster 中，由于数据分散在多个节点，**跨节点操作会受到限制**：

| 操作类型 | 单节点 | 跨节点 | Hash Tag 解决 |
|---------|--------|--------|--------------|
| **MGET/MSET** | ✅ | ❌ CROSSSLOT | ✅ |
| **事务（MULTI）** | ✅ | ❌ CROSSSLOT | ✅ |
| **Lua 脚本** | ✅ | ❌ CROSSSLOT | ✅ |
| **SUNION/SINTER** | ✅ | ❌ CROSSSLOT | ✅ |
| **ZINTERSTORE** | ✅ | ❌ CROSSSLOT | ✅ |

```bash
# ❌ 错误示例：跨节点操作
127.0.0.1:6379> MGET user:1:name user:2:name user:3:name
(error) CROSSSLOT Keys in request don't hash to the same slot

# ✅ 正确示例：使用 Hash Tag
127.0.0.1:6379> MGET user:{1}:name user:{1}:age user:{1}:email
1) "Alice"
2) "25"
3) "alice@example.com"
```

##### 📝 Hash Tag 怎么用？

**场景1：用户数据聚合**

```java
/**
 * 场景：电商系统，需要批量获取用户信息
 */
public class UserService {
    
    @Autowired
    private RedisTemplate<String, String> redisTemplate;
    
    /**
     * 保存用户信息（使用 Hash Tag）
     */
    public void saveUser(Long userId, User user) {
        String tag = String.valueOf(userId);
        
        // 所有 key 都使用 {userId} 作为 Hash Tag
        Map<String, String> userData = new HashMap<>();
        userData.put("user:{" + tag + "}:name", user.getName());
        userData.put("user:{" + tag + "}:age", String.valueOf(user.getAge()));
        userData.put("user:{" + tag + "}:email", user.getEmail());
        userData.put("user:{" + tag + "}:phone", user.getPhone());
        
        // 批量写入（单次操作，性能高）
        redisTemplate.opsForValue().multiSet(userData);
    }
    
    /**
     * 批量获取用户信息（使用 MGET）
     */
    public User getUser(Long userId) {
        String tag = String.valueOf(userId);
        
        // 批量读取（单次操作，性能高）
        List<String> keys = Arrays.asList(
            "user:{" + tag + "}:name",
            "user:{" + tag + "}:age",
            "user:{" + tag + "}:email",
            "user:{" + tag + "}:phone"
        );
        
        List<String> values = redisTemplate.opsForValue().multiGet(keys);
        
        return User.builder()
            .name(values.get(0))
            .age(Integer.parseInt(values.get(1)))
            .email(values.get(2))
            .phone(values.get(3))
            .build();
    }
}
```

**场景2：订单事务操作**

```java
/**
 * 场景：订单系统，需要在事务中操作多个 key
 */
public class OrderService {
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    /**
     * 创建订单（使用事务）
     */
    public void createOrder(Long orderId, Order order) {
        String tag = String.valueOf(orderId);
        
        // 使用事务保证原子性
        redisTemplate.execute(new SessionCallback<Object>() {
            @Override
            public Object execute(RedisOperations operations) throws DataAccessException {
                // 开启事务
                operations.multi();
                
                // 所有操作的 key 都在同一个 slot
                operations.opsForValue().set("order:{" + tag + "}:status", "PENDING");
                operations.opsForValue().set("order:{" + tag + "}:amount", order.getAmount());
                operations.opsForValue().decrement("product:{" + tag + "}:stock", order.getQuantity());
                operations.opsForValue().increment("user:{" + tag + "}:order_count");
                
                // 提交事务
                return operations.exec();
            }
        });
    }
}
```

```java
/**
 * 场景：秒杀系统，使用 Lua 脚本保证原子性
 */
public class SeckillService {
    
    @Autowired
    private RedisTemplate<String, String> redisTemplate;
    
    /**
     * 秒杀抢购（使用 Lua 脚本）
     */
    public boolean seckill(Long activityId, Long userId) {
        String tag = String.valueOf(activityId);
        
        // Lua 脚本：检查库存 + 扣减库存 + 记录用户
        String script = 
            "local stock = redis.call('GET', KEYS[1]) " +
            "if tonumber(stock) > 0 then " +
            "  redis.call('DECR', KEYS[1]) " +
            "  redis.call('SADD', KEYS[2], ARGV[1]) " +
            "  return 1 " +
            "else " +
            "  return 0 " +
            "end";
        
        // 所有 key 都使用相同的 Hash Tag
        List<String> keys = Arrays.asList(
            "seckill:{" + tag + "}:stock",      // 库存
            "seckill:{" + tag + "}:users"       // 已购用户集合
        );
        
        Long result = redisTemplate.execute(
            new DefaultRedisScript<>(script, Long.class),
            keys,
            String.valueOf(userId)
        );
        
        return result != null && result == 1;
    }
}
```

##### 🎬 什么场景下有用？

| 场景 | 是否需要 Hash Tag | 原因 |
|------|------------------|------|
| **批量查询用户信息** | ✅ 推荐 | MGET 性能高，减少网络开销 |
| **订单事务操作** | ✅ 必须 | 事务要求所有 key 在同一节点 |
| **Lua 脚本原子操作** | ✅ 必须 | Lua 脚本要求所有 key 在同一节点 |
| **秒杀库存扣减** | ✅ 推荐 | 保证原子性，避免超卖 |
| **集合交并运算** | ✅ 必须 | SINTER/SUNION 要求同节点 |
| **独立的缓存数据** | ❌ 不需要 | 数据无关联，分散存储更好 |
| **简单的 GET/SET** | ❌ 不需要 | 单 key 操作无需 Hash Tag |

##### ⚠️ Hash Tag 使用注意事项

**1. 数据倾斜问题**

```
❌ 错误示例：所有用户使用相同的 Hash Tag
user:{common}:1:name
user:{common}:2:name
user:{common}:3:name
...
user:{common}:1000000:name

问题：
- 所有数据都在同一个 slot
- 单个节点压力过大
- 失去了 Cluster 分片的意义

✅ 正确示例：按用户 ID 分组
user:{1}:name      → Slot 5970
user:{2}:name      → Slot 8456
user:{3}:name      → Slot 12345
...

优势：
- 数据均匀分布在各个节点
- 同一用户的数据在同一节点（可以使用 MGET/事务）
```

**2. 合理设计 Hash Tag**

```java
// ✅ 推荐：按业务实体分组
"user:{userId}:profile"
"user:{userId}:settings"
"order:{orderId}:detail"
"order:{orderId}:items"

// ❌ 不推荐：过度使用 Hash Tag
"cache:{all}:user:1"      // 所有缓存在一个节点
"session:{global}:xxx"    // 所有会话在一个节点

// ✅ 推荐：按时间分片
"log:{2026-01-28}:error"  // 按天分片
"stats:{2026-01}:pv"      // 按月分片
```

**3. 性能对比**

```
场景：获取 100 个用户的姓名

方案1：不使用 Hash Tag（跨节点）
- 100 次 GET 请求
- 网络往返：100 次
- 耗时：~100ms

方案2：使用 Hash Tag（同节点）
- 1 次 MGET 请求（100 个 key）
- 网络往返：1 次
- 耗时：~5ms

性能提升：20 倍 ⚡
```

**4. 监控与告警**

```bash
# 检查 slot 分布是否均匀
redis-cli --cluster check 192.168.1.101:6379

# 查看每个节点的 key 数量
redis-cli -c -h 192.168.1.101 -p 6379 INFO keyspace

# 如果某个节点 key 数量远超其他节点，可能存在数据倾斜
# 需要重新设计 Hash Tag 策略
```

##### 📊 Hash Tag 最佳实践总结

```
✅ 适合使用 Hash Tag 的场景：
1. 需要批量操作（MGET/MSET）
2. 需要事务保证（MULTI/EXEC）
3. 需要 Lua 脚本原子性
4. 需要集合运算（SINTER/SUNION）
5. 业务上有强关联的数据

❌ 不适合使用 Hash Tag 的场景：
1. 数据量巨大且无关联
2. 独立的缓存数据
3. 需要均匀分布的数据
4. 简单的 GET/SET 操作

设计原则：
1. Hash Tag 粒度要合理（不要太粗也不要太细）
2. 避免数据倾斜（监控各节点 key 数量）
3. 优先考虑业务需求（性能 vs 分布均匀）
4. 定期检查 slot 分布情况
```

### 4.2 客户端路由：MOVED 与 ASK 重定向

#### 🔀 MOVED 重定向：槽位永久迁移

```
场景：集群扩容，Slot 13320 从 Master2 迁移到 Master4

客户端请求 → Master2 → 返回 MOVED 13320 192.168.1.104:6379
客户端更新映射表 → 重新请求 Master4 → 返回数据 ✅

关键：MOVED 命令会更新客户端缓存的槽位映射表
```

#### ⏳ ASK 重定向：槽位临时迁移

```
场景：Slot 13320 正在从 Master2 迁移到 Master4（部分数据已迁移）

客户端请求 → Master2 → 返回 ASK 13320 192.168.1.104:6379
客户端发送 ASKING → Master4 → 返回数据 ✅

关键：ASK 命令不会更新客户端缓存，每次请求都可能需要重定向
```

### 4.3 节点通信：Gossip 协议

Redis Cluster 节点间采用 **Gossip 协议**进行通信：
- 每个节点定期随机选择几个节点发送 PING 消息
- 包含自己的状态和部分其他节点的状态
- 信息逐渐扩散到整个集群

**Gossip 消息类型**：
- **MEET**：用于加入集群
- **PING**：心跳消息，每秒随机选择 5 个节点
- **PONG**：响应 PING 或 MEET
- **FAIL**：通知某个节点已下线

### 4.4 故障转移：自动选举

#### 🚨 网络抖动问题

配置 `cluster-node-timeout 15000`（15秒）可以避免网络抖动导致的频繁主从切换。

#### 🗳️ 故障转移流程

```
1. Slave 发现自己的 Master 变为 FAIL 状态
2. 将自己记录的集群 currentEpoch + 1
3. 广播 FAILOVER_AUTH_REQUEST 信息（请求投票）
4. 其他 Master 节点响应，判断请求者的合法性，发送 FAILOVER_AUTH_ACK
5. Slave 收到超过半数 Master 的 ACK 后变成新 Master
6. Slave 广播 Pong 消息通知其他集群节点
```

**为什么至少需要 3 个 Master？**
- 2 个 Master：挂 1 个后，只有 1 票，无法达到半数 ❌
- 3 个 Master：挂 1 个后，有 2 票，超过半数 ✅

**为什么推荐奇数个 Master？**
- 3 个和 4 个容错能力相同（都是 1 个）
- 3 个节省 1 台机器，更经济 ✅

#### ⏱️ 选举延迟机制

```
延迟计算公式：
DELAY = 500ms + random(0~500ms) + SLAVE_RANK * 1000ms

SLAVE_RANK：根据复制偏移量排名
- 偏移量最大（数据最新）：RANK = 0，延迟最短
- 偏移量第二：RANK = 1
- 偏移量第三：RANK = 2

目的：确保数据最新的 Slave 优先当选，减少数据丢失
```

#### 🚨 多个 Master 同时宕机场景分析

> **核心问题**：3主3从 Cluster 架构中，如果同时挂掉 2 个 Master 会怎么样？这是生产环境中必须了解的极端故障场景。

**场景1：挂掉的 2 个 Master 都有 Slave（最常见）**

```
初始架构（3主3从）：

Master1 (Slot: 0-5460)  ──┐
  └─ Slave1             ──┼── 正常运行
                          │
Master2 (Slot: 5461-10922)──┼── ❌ 宕机
  └─ Slave2             ──┼── ✅ 存活
                          │
Master3 (Slot: 10923-16383)─┼── ❌ 宕机
  └─ Slave3             ──┼── ✅ 存活
                          │
                          └── 集群状态：？

故障转移流程：

T0: Master2 和 Master3 同时宕机
    ├─ 集群检测到 2 个 Master 下线
    └─ 触发故障转移流程

T1: Slave2 和 Slave3 开始选举（约 1-2 秒）
    ├─ Slave2 向 Master1 请求投票
    ├─ Slave3 向 Master1 请求投票
    └─ Master1 分别投票给 Slave2 和 Slave3

T2: 选举完成（约 15-20 秒）
    ├─ Slave2 升级为新的 Master2'
    ├─ Slave3 升级为新的 Master3'
    └─ 集群恢复正常

最终架构：

Master1 (Slot: 0-5460)      ✅ 正常
  └─ 无 Slave               ⚠️ 无备份

Master2' (Slot: 5461-10922) ✅ 正常（原 Slave2）
  └─ 无 Slave               ⚠️ 无备份

Master3' (Slot: 10923-16383)✅ 正常（原 Slave3）
  └─ 无 Slave               ⚠️ 无备份

结果：
✅ 集群继续提供服务（所有 slot 都有 Master）
✅ 数据不丢失（Slave 有完整数据）
⚠️ 高可用性降低（所有 Master 都没有备份）
⚠️ 如果再挂 1 个 Master，集群将不可用

关键点：
1. 故障转移时间：15-20 秒（两个 Slave 并行选举）
2. 数据完整性：完整（假设复制延迟很小）
3. 集群可用性：可用（所有 slot 都有 Master）
4. 风险等级：高（无备份，不能再挂任何 Master）
```

**场景2：挂掉的 2 个 Master 中有 1 个没有 Slave（危险）**

```
初始架构（3主3从）：

Master1 (Slot: 0-5460)  ──┐
  └─ Slave1             ──┼── 正常运行
                          │
Master2 (Slot: 5461-10922)──┼── ❌ 宕机
  └─ Slave2             ──┼── ✅ 存活
                          │
Master3 (Slot: 10923-16383)─┼── ❌ 宕机
  └─ 无 Slave（已经挂了）──┼── ❌ 无备份
                          │
                          └── 集群状态：？

故障转移流程：

T0: Master2 和 Master3 同时宕机
    ├─ Master2 有 Slave2 可以故障转移
    └─ Master3 无 Slave，无法故障转移 ⚠️

T1: Slave2 开始选举（约 1-2 秒）
    ├─ Slave2 向 Master1 请求投票
    └─ Master1 投票给 Slave2

T2: Slave2 升级为 Master2'（约 15 秒）
    └─ Master3 的 slot (10923-16383) 无人负责 ❌

最终架构：

Master1 (Slot: 0-5460)      ✅ 正常
  └─ Slave1                 ✅ 有备份

Master2' (Slot: 5461-10922) ✅ 正常（原 Slave2）
  └─ 无 Slave               ⚠️ 无备份

Master3 (Slot: 10923-16383) ❌ 宕机，无 Slave
  └─ slot 10923-16383 不可用 ❌

结果：
❌ 集群不可用（默认配置 cluster-require-full-coverage yes）
❌ 部分 slot 无法提供服务（10923-16383）
❌ 所有操作都会失败（即使访问 slot 0-10922）

关键配置：cluster-require-full-coverage

# cluster-require-full-coverage yes（默认）
├─ 任何 slot 不可用，整个集群拒绝服务 ❌
└─ 保证数据完整性，但牺牲可用性

# cluster-require-full-coverage no（推荐生产环境）
├─ 部分 slot 不可用，其他 slot 继续服务 ✅
└─ 提高可用性，但部分数据不可访问

如果配置 cluster-require-full-coverage no：
✅ slot 0-10922 继续提供服务
❌ slot 10923-16383 不可用
⚠️ 访问不可用 slot 的 key 会返回 CLUSTERDOWN 错误
```

**场景3：挂掉的 2 个 Master 都没有 Slave（灾难）**

```
初始架构（异常情况，不应该出现）：

Master1 (Slot: 0-5460)      ✅ 正常
  └─ Slave1                 ✅ 有备份

Master2 (Slot: 5461-10922)  ❌ 宕机

❌ 集群完全不可用（2/3 的 slot 无法提供服务）
❌ 即使配置 cluster-require-full-coverage no，也只有 1/3 的数据可访问
❌ 数据永久丢失（Master2 和 Master3 的数据无法恢复）

这种情况通常是因为：
1. Slave 节点已经挂了，但没有及时修复

预防措施：
```

**场景4：挂掉的 2 个 Master 是同一个 slot 范围的主从（极端情况）**

```
初始架构（3主3从）：

  └─ Slave1                 ❌ 同时宕机（主从同时挂）

  └─ Slave2                 ✅ 正常

  └─ Slave3                 ✅ 正常

结果：
❌ slot 0-5460 完全不可用（主从都挂了）
✅ slot 5461-16383 正常提供服务

✅ 2/3 的数据可以访问
❌ 1/3 的数据不可访问
⚠️ 访问 slot 0-5460 的 key 会返回 CLUSTERDOWN 错误

这种情况通常是因为：
1. 主从部署在同一台物理机（反模式 ❌）
2. 主从部署在同一个机架（电源故障）
3. 主从部署在同一个交换机下（网络故障）

预防措施：
✅ 主从必须部署在不同物理机
✅ 主从必须部署在不同机架
✅ 主从必须部署在不同交换机下
```

### 4.5 集群脑裂：数据丢失问题

#### 🧠 什么是脑裂？

网络分区导致集群分裂成多个部分，出现多个 Master 对外提供写服务。网络恢复后，其中一个 Master 降级为 Slave，导致大量数据丢失。

#### 🛡️ 解决方案：min-replicas-to-write

```bash
# redis.conf 配置
min-replicas-to-write 1      # 写数据成功最少同步的 Slave 数量
min-replicas-max-lag 10      # Slave 复制延迟不能超过 10 秒

# 原理：Master 必须有足够的 Slave 在线才能写入
# 3 个 Master 的集群：配置为 1（1 + 1 = 2 > 3/2）
# 5 个 Master 的集群：配置为 2（2 + 1 = 3 > 5/2）

# 注意：这个配置会影响集群可用性，需要权衡
```

### 4.6 Cluster 配置与创建

```bash
# redis.conf 配置
cluster-enabled yes                    # 开启集群模式
cluster-config-file nodes-6379.conf    # 集群配置文件（自动生成）

cluster-require-full-coverage no       # 部分槽不可用时，其他槽继续服务

# 创建集群（推荐方式）
redis-cli --cluster create \
  192.168.1.101:6379 192.168.1.102:6379 192.168.1.103:6379 \

```

### 4.7 Cluster 读写模式：主写从读还是主读主写？

#### 🔍 核心问题

**Redis Cluster 默认是"主读主写"模式，不是"主写从读"！**

这是一个非常重要的概念，很多人会误以为 Cluster 像主从复制一样自动实现读写分离。

#### 📊 Cluster 默认读写模式

```
默认模式：主读主写（Master 处理所有读写请求）

    ┌─────────────────┐
    │   Master 1      │ ← 读请求 ✅
    │ Slot: 0-5460    │ ← 写请求 ✅
    └────────┬────────┘
             │ 复制
    ┌────────▼────────┐
    │   Slave 1       │ ← 读请求 ❌（默认不处理）
    │ 备份 Master1    │ ← 写请求 ❌
    └─────────────────┘

原因：
1. Cluster 的设计目标是"数据分片 + 高可用"，不是读写分离
2. 客户端根据 slot 路由到对应的 Master 节点
3. Slave 节点主要用于故障转移，不参与读请求
```

#### 🎯 为什么 Cluster 默认不读从节点？

| 原因 | 说明 |
|------|------|
| **数据一致性** | 主从复制是异步的，从节点可能有延迟，读到旧数据 |
| **路由复杂性** | 客户端需要维护 slot → Master/Slave 的映射关系 |
| **设计理念** | Cluster 通过增加 Master 节点来提升性能，而不是增加 Slave |

#### ⚙️ 如何在 Cluster 中实现读写分离？

虽然不推荐，但 Redis Cluster 确实支持从 Slave 读取数据，需要使用 **READONLY** 命令。

```bash
# 连接到 Slave 节点
redis-cli -c -h 192.168.1.104 -p 6379

# 开启只读模式
127.0.0.1:6379> READONLY
OK

# 现在可以从 Slave 读取数据
127.0.0.1:6379> GET user:123:name
"Alice"

# 关闭只读模式
127.0.0.1:6379> READWRITE
OK
```

#### 💻 Java 客户端实现 Cluster 读写分离

```java
/**
 * Jedis 实现 Cluster 读写分离
 */
public class RedisClusterConfig {
    
    @Bean
    public JedisCluster jedisCluster() {
        Set<HostAndPort> nodes = new HashSet<>();
        nodes.add(new HostAndPort("192.168.1.101", 6379));
        nodes.add(new HostAndPort("192.168.1.102", 6379));
        nodes.add(new HostAndPort("192.168.1.103", 6379));
        
        JedisPoolConfig poolConfig = new JedisPoolConfig();
        poolConfig.setMaxTotal(100);
        poolConfig.setMaxIdle(20);
        
        return new JedisCluster(nodes, 2000, 2000, 5, poolConfig);
    }
}

/**
 * Lettuce 实现 Cluster 读写分离（推荐）
 */
@Configuration
public class LettuceClusterConfig {
    
    @Bean
    public LettuceConnectionFactory redisConnectionFactory() {
        // 集群节点配置
        RedisClusterConfiguration clusterConfig = new RedisClusterConfiguration();
        clusterConfig.addClusterNode(new RedisNode("192.168.1.101", 6379));
        clusterConfig.addClusterNode(new RedisNode("192.168.1.102", 6379));
        clusterConfig.addClusterNode(new RedisNode("192.168.1.103", 6379));
        
        // 开启读写分离：从 Slave 读取
        ClusterTopologyRefreshOptions topologyRefreshOptions = ClusterTopologyRefreshOptions.builder()
            .enablePeriodicRefresh(Duration.ofMinutes(10))  // 定期刷新拓扑
            .enableAllAdaptiveRefreshTriggers()             // 自适应刷新
            .build();
        
        ClusterClientOptions clientOptions = ClusterClientOptions.builder()
            .topologyRefreshOptions(topologyRefreshOptions)
            .build();
        
        LettuceClientConfiguration clientConfig = LettuceClientConfiguration.builder()
            .readFrom(ReadFrom.REPLICA_PREFERRED)  // 优先从 Slave 读取 ⭐⭐⭐
            .clientOptions(clientOptions)
            .build();
        
        return new LettuceConnectionFactory(clusterConfig, clientConfig);
    }
}
```

#### 📋 ReadFrom 策略说明

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| **MASTER** | 只从 Master 读（默认） | 强一致性要求 |
| **MASTER_PREFERRED** | 优先 Master，Master 不可用时读 Slave | 一般场景 |
| **REPLICA** | 只从 Slave 读 | 读多写少，可容忍延迟 |
| **REPLICA_PREFERRED** | 优先 Slave，Slave 不可用时读 Master | 读多写少（推荐）⭐ |
| **LOWEST_LATENCY** | 选择延迟最低的节点 | 性能优先 |
| **ANY** | 任意节点 | 极端高并发 |

### 4.8 读多写少场景：增加从节点还是增加主节点？

#### 🤔 核心问题

**在读多写少的场景下，应该增加从节点（3主6从）还是增加主节点（6主6从）？**

#### 📊 方案对比

**方案1：增加从节点（3主6从）**

```
架构：3 个 Master + 6 个 Slave

    Master1 (Slot 0-5460)
       ├─ Slave1-1
       └─ Slave1-2
    
    Master2 (Slot 5461-10922)
       ├─ Slave2-1
       └─ Slave2-2
    
    Master3 (Slot 10923-16383)
       ├─ Slave3-1
       └─ Slave3-2

优势：
✅ 每个 Master 有 2 个 Slave 备份，可用性更高
✅ 可以从 Slave 读取，分担 Master 读压力（需要开启 READONLY）
✅ 故障转移更快（有更多备份节点）

劣势：
❌ 写性能没有提升（仍然只有 3 个 Master）
❌ 数据容量没有增加（仍然受限于单个 Master）
❌ 需要客户端支持读写分离（增加复杂度）
❌ 可能读到延迟数据（主从复制延迟）
```

**方案2：增加主节点（6主6从）**

```
架构：6 个 Master + 6 个 Slave

    Master1 (Slot 0-2730)      Master4 (Slot 8192-10922)
       └─ Slave1                  └─ Slave4
    
    Master2 (Slot 2731-5460)   Master5 (Slot 10923-13653)
       └─ Slave2                  └─ Slave5
    
    Master3 (Slot 5461-8191)   Master6 (Slot 13654-16383)
       └─ Slave3                  └─ Slave6

优势：
✅ 读性能提升 2 倍（6 个 Master 处理读请求）
✅ 写性能提升 2 倍（6 个 Master 处理写请求）
✅ 数据容量提升 2 倍（6 个 Master 存储数据）
✅ 客户端无需改动（自动路由到对应 Master）
✅ 数据强一致性（直接读 Master）

劣势：
❌ 成本更高（需要更多高性能服务器）
❌ 管理复杂度增加（更多节点）
```

#### 🎯 推荐方案

| 场景 | 推荐方案 | 原因 |
|------|----------|------|
| **读多写少，可容忍延迟** | 3主6从 + 读写分离 | 成本低，充分利用 Slave |
| **读多写少，强一致性** | 6主6从 | 读 Master，无延迟 |
| **写压力大** | 6主6从 | 增加 Master 提升写性能 |
| **数据量大** | 6主6从 | 增加 Master 提升容量 |
| **高可用要求极高** | 3主6从 | 每个 Master 有 2 个备份 |

#### ⚠️ 增加从节点的潜在问题

**1. 主从复制延迟**

```
问题：
- 主从复制是异步的，Slave 数据可能落后 Master
- 从节点越多，Master 的复制压力越大

场景：
1. 用户修改密码（写入 Master）
2. 立即用新密码登录（读取 Slave）
3. Slave 数据未同步，登录失败 ❌

解决方案：
- 关键数据必须读 Master
- 使用 WAIT 命令确保同步：WAIT 2 1000（等待 2 个 Slave 同步，超时 1 秒）
```

**2. 网络带宽压力**

```
问题：
- 每个 Slave 都需要从 Master 全量复制数据
- 3 个 Master + 6 个 Slave = 每个 Master 需要向 2 个 Slave 复制

计算：
- Master 数据量：10GB
- 全量复制时间：10GB / 100MB/s = 100 秒
- 网络带宽占用：100MB/s × 2 = 200MB/s（每个 Master）

影响：
- 全量复制期间，Master 性能下降
- 网络带宽占用高
```

**3. Gossip 协议开销**

```
问题：
- Cluster 节点通过 Gossip 协议通信
- 节点越多，心跳消息越多

计算：
- 9 个节点（3主6从）：每个节点维护 8 个连接
- 心跳频率：每秒 1 次
- 消息大小：约 1KB
- 总带宽：9 × 8 × 1KB = 72KB/s（可接受）

- 100 个节点：每个节点维护 99 个连接
- 总带宽：100 × 99 × 1KB = 9.9MB/s（较大）

建议：
- 节点数不超过 1000 个（官方推荐）
- 3主6从（9 个节点）完全没问题
```

**4. 故障转移时间增加**

```
问题：
- 从节点越多，选举时需要更多投票
- 可能导致故障转移时间略微增加

影响：
- 3主3从：故障转移约 15 秒
- 3主6从：故障转移约 15-20 秒（影响不大）
```

### 4.10 Redis Cluster vs MySQL 主从架构对比

#### 🔄 架构设计对比

**Redis Cluster 架构**

```
Redis Cluster（分片 + 高可用）

    Master1 (Slot 0-5460)      Master2 (Slot 5461-10922)    Master3 (Slot 10923-16383)
         └─ Slave1                    └─ Slave2                     └─ Slave3

特点：
✅ 数据分片：每个 Master 只存储部分数据（按 slot 分配）
✅ 写扩展：增加 Master 可提升写性能
✅ 容量扩展：增加 Master 可提升总容量
✅ 自动路由：客户端自动路由到对应节点
✅ 无中心化：所有节点地位平等
```

**MySQL 主从架构**

```
MySQL 主从复制（读写分离）

         Master (写)
            │
    ┌───────┼───────┐
    │       │       │
  Slave1  Slave2  Slave3 (读)

特点：
✅ 读扩展：增加 Slave 可提升读性能
❌ 写瓶颈：所有写操作都在 Master，无法扩展
❌ 容量瓶颈：所有节点存储全量数据，容量受限于单机
❌ 需要中间件：需要 ProxySQL/MyCat 做读写分离路由
```

#### 📊 核心差异对比

| 维度 | Redis Cluster | MySQL 主从 |
|------|---------------|------------|
| **数据分布** | 分片存储（每个节点存部分数据） | 全量复制（每个节点存全部数据） |
| **写性能扩展** | ✅ 增加 Master 可线性扩展 | ❌ 无法扩展（单 Master 瓶颈） |
| **读性能扩展** | ✅ 增加 Master 或开启从节点读 | ✅ 增加 Slave 可扩展 |
| **容量扩展** | ✅ 增加 Master 可线性扩展 | ❌ 受限于单机磁盘容量 |
| **路由方式** | 客户端智能路由（CRC16 计算） | 需要中间件或应用层路由 |
| **故障转移** | 自动（Raft 选举） | 需要 MHA/Orchestrator |
| **数据一致性** | 最终一致性（异步复制） | 最终一致性（异步复制） |
| **跨节点事务** | ❌ 不支持（除非 Hash Tag） | ✅ 支持（单 Master） |
| **适用场景** | 缓存、高并发读写 | 持久化存储、事务 |

#### 🎯 性能扩展方案对比

**场景：读多写少，需要提升性能**

##### Redis 扩展方案

**方案1：增加 Master 节点（推荐）⭐⭐⭐⭐⭐**

```
3主3从 → 6主6从

优势：
✅ 读性能提升 2 倍（6 个 Master 处理读）
✅ 写性能提升 2 倍（6 个 Master 处理写）
✅ 容量提升 2 倍
✅ 客户端无需改动（自动路由）
✅ 数据强一致性（读 Master）

操作：
1. 添加 3 个新 Master 节点
2. 重新分配 slot（reshard）
3. 为新 Master 添加 Slave
4. 验证集群状态

适用：写压力大、数据量大、需要强一致性
```

**方案2：增加 Slave + 读写分离**

```
3主3从 → 3主6从

优势：
✅ 读性能提升（从 Slave 读取）
✅ 成本较低（只增加 Slave）
✅ 可用性提升（更多备份）

劣势：
❌ 写性能不变
❌ 容量不变
❌ 可能读到延迟数据
❌ 需要客户端支持（Lettuce ReadFrom）

操作：
1. 添加 3 个新 Slave 节点
2. 配置客户端读写分离
3. 监控复制延迟

适用：读多写少、可容忍延迟、成本敏感
```

##### MySQL 扩展方案

**方案1：增加从库（读写分离）⭐⭐⭐**

```
1主1从 → 1主3从

✅ 读性能提升（多个 Slave 分担读请求）
✅ 实现简单

劣势：
❌ 写性能不变（单 Master 瓶颈）
❌ 容量不变（受限于单机）
❌ 主从延迟问题
❌ 需要读写分离中间件（ShardingSphere、MyCat）

操作：
1. 搭建新的 Slave 节点
2. 配置主从复制
3. 部署读写分离中间件
4. 应用层配置数据源

```

```
单库 → 多库分片（类似 Redis Cluster）

架构：
DB1 (用户 ID % 4 = 0)    DB2 (用户 ID % 4 = 1)
  └─ Slave1                └─ Slave2

  └─ Slave3                └─ Slave4

优势：
✅ 读性能线性扩展
✅ 写性能线性扩展（多个 Master）
✅ 容量线性扩展
✅ 类似 Redis Cluster 的分片思想

❌ 跨库事务困难
❌ 跨库 JOIN 困难
❌ 数据迁移复杂

操作：
1. 设计分片键（如 user_id）
2. 部署多个 MySQL 实例
3. 配置 ShardingSphere 分片规则
4. 数据迁移（全量 + 增量）
5. 应用层改造

适用：写压力大、数据量大（> 1TB）、可接受复杂度
```

**方案3：垂直拆分**

```
单库 → 多个业务库

原架构：
order_db (订单表 + 用户表 + 商品表)

拆分后：
order_db (订单表)
user_db (用户表)
product_db (商品表)

优势：
✅ 降低单库压力
✅ 业务隔离
✅ 便于独立扩展

劣势：
❌ 跨库 JOIN 困难
❌ 分布式事务问题

适用：业务复杂、表很多、不同业务压力不均
```

```
@Service
public class OrderService {
    @Autowired
    private RedisTemplate redisTemplate;
    
    @Autowired
    private OrderMapper orderMapper;

    public Order getOrder(Long orderId) {
        // 1. 先查 Redis
        String cacheKey = "order:" + orderId;
        Order order = (Order) redisTemplate.opsForValue().get(cacheKey);
        
        if (order != null) {
            return order;  // 缓存命中

        // 2. 缓存未命中，查 MySQL
        order = orderMapper.selectById(orderId);
        
        // 3. 写入 Redis（过期时间 1 小时）
        if (order != null) {
            redisTemplate.opsForValue().set(cacheKey, order, 1, TimeUnit.HOURS);
        }
        
        return order;
    }
    
    // 创建订单（先写 MySQL，再删缓存）
    @Transactional
    public void createOrder(Order order) {
        // 1. 写入 MySQL
        orderMapper.insert(order);
        
        // 2. 删除缓存（保证一致性）
        String cacheKey = "order:" + order.getId();
        redisTemplate.delete(cacheKey);
    }
}
```

### 4.11 Redis Cluster vs Kafka 副本机制深度对比

> **为什么对比 Redis 和 Kafka？**
> 
> Redis Cluster 和 Kafka 都是分布式系统，都使用了分片（Partition/Slot）和副本（Replica）机制来实现高可用和高性能。但它们的设计理念和实现方式有本质区别，深入理解这些差异有助于在实际项目中做出正确的技术选型。

#### 🏗️ 架构设计对比

**Redis Cluster 架构**

```
Redis Cluster（分片 + 主从复制）

    ┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐
    │   Master 1      │  │   Master 2      │  │   Master 3      │
    │ Slot: 0-5460    │  │ Slot: 5461-     │  │ Slot: 10923-    │
    │ (主节点)        │  │ 10922 (主节点)  │  │ 16383 (主节点)  │
    └────────┬────────┘  └────────┬────────┘  └────────┬────────┘
             │                    │                    │
             │ 异步复制           │ 异步复制           │ 异步复制
             │                    │                    │
    ┌────────▼────────┐  ┌────────▼────────┐  ┌────────▼────────┐
    │   Slave 1       │  │   Slave 2       │  │   Slave 3       │
    │ (从节点)        │  │ (从节点)        │  │ (从节点)        │
    └─────────────────┘  └─────────────────┘  └─────────────────┘

核心特点：
✅ 主从模式：Master 负责读写，Slave 只做备份（默认不读）
✅ 异步复制：Master 写入后立即返回，异步同步到 Slave
✅ 自动故障转移：Master 宕机，Slave 自动提升为 Master
✅ 客户端路由：客户端根据 CRC16(key) % 16384 计算 slot，直连对应节点
✅ 无中心化：所有节点地位平等，通过 Gossip 协议通信
```

**Kafka 架构**

```
Kafka（分区 + 副本同步）

Topic: order_topic (3 个 Partition，3 个副本)

Partition 0:
    ┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐
    │ Leader (Broker1)│  │ Follower (Broker2)│ │ Follower (Broker3)│
    │ 负责读写        │  │ 同步复制        │  │ 同步复制        │
    └─────────────────┘  └─────────────────┘  └─────────────────┘
             │                    ▲                    ▲
             │                    │                    │
             └────────────────────┴────────────────────┘
                        同步复制（ISR 机制）

Partition 1:
    ┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐
    │ Leader (Broker2)│  │ Follower (Broker3)│ │ Follower (Broker1)│
    └─────────────────┘  └─────────────────┘  └─────────────────┘

Partition 2:
    ┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐
    │ Leader (Broker3)│  │ Follower (Broker1)│ │ Follower (Broker2)│
    └─────────────────┘  └─────────────────┘  └─────────────────┘

核心特点：
✅ Leader-Follower 模式：Leader 负责读写，Follower 同步复制
✅ 同步复制（可配置）：支持 acks=all，等待所有 ISR 副本确认
✅ ISR 机制：In-Sync Replicas，保证数据一致性
✅ 自动故障转移：Leader 宕机，从 ISR 中选举新 Leader
✅ 中心化协调：依赖 ZooKeeper/KRaft 管理元数据和选举
```

#### 📊 核心差异对比

| 维度 | Redis Cluster | Kafka |
|------|---------------|-------|
| **设计目标** | 高性能缓存、低延迟 KV 存储 | 高吞吐消息队列、日志存储 |
| **数据模型** | Key-Value（String、Hash、List 等） | 消息流（Partition + Offset） |
| **分片单位** | Slot（16384 个） | Partition（自定义数量） |
| **副本角色** | Master（读写）+ Slave（备份） | Leader（读写）+ Follower（同步） |
| **复制方式** | 异步复制（无法配置同步） | 同步/异步可配置（acks=0/1/all） |
| **数据一致性** | 最终一致性（可能丢数据） | 强一致性（acks=all + ISR） |
| **故障转移** | 自动（Raft 选举，10-30秒） | 自动（ZooKeeper/KRaft 选举，秒级） |
| **读操作** | 默认只读 Master（可配置读 Slave） | 只读 Leader（Follower 不提供读） |
| **写操作** | 只写 Master | 只写 Leader |
| **持久化** | 可选（RDB/AOF/混合） | 强制（磁盘日志） |
| **数据丢失风险** | 高（异步复制 + 可选持久化） | 低（同步复制 + 强制持久化） |
| **延迟** | 极低（< 1ms） | 低（1-10ms） |
| **吞吐量** | 高（10万+ QPS） | 极高（百万+ TPS） |
| **协调机制** | Gossip 协议（无中心） | ZooKeeper/KRaft（中心化） |
| **客户端路由** | 智能路由（客户端计算 slot） | 客户端缓存元数据（Broker 列表） |
| **扩展方式** | 增加节点 + Reshard | 增加 Broker + 重新分配 Partition |
| **适用场景** | 缓存、会话、限流、排行榜 | 消息队列、日志收集、流处理 |

#### 🔬 副本机制深度对比

**1. 复制方式对比**

```
┌─────────────────────────────────────────────────────────────────┐
│ Redis Cluster：异步复制（无法配置同步）                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│ 客户端写入流程：                                                  │
│                                                                  │
│ 1. 客户端发送：SET user:1 "Alice"                               │
│    └─ 客户端计算 slot：CRC16("user:1") % 16384 = 5000          │
│    └─ 路由到 Master 1（负责 slot 0-5460）                       │
│                                                                  │
│ 2. Master 1 处理：                                               │
│    ├─ 写入内存                                                   │
│    ├─ 立即返回 OK 给客户端 ✅（不等待 Slave）                    │
│    └─ 异步复制到 Slave 1                                        │
│                                                                  │
│ 3. Slave 1 处理：                                                │
│    ├─ 接收复制命令                                               │
│    ├─ 写入内存                                                   │
│    └─ 发送 ACK 给 Master（Master 不等待）                       │
│                                                                  │
│ 时间线：                                                          │
│ T0: 客户端发送 SET                                               │
│ T1: Master 写入内存（1ms）                                       │
│ T2: Master 返回 OK（客户端收到响应）✅                           │
│ T3: Master 异步复制到 Slave（2-5ms）                            │
│ T4: Slave 写入完成                                               │
│                                                                  │
│ 风险：                                                            │
│ ❌ 如果 Master 在 T2-T4 之间宕机，数据丢失                       │
│ ❌ 客户端认为写入成功，但 Slave 没有数据                         │
│ ❌ 无法配置同步复制                                              │
│                                                                  │
│ 优势：                                                            │
│ ✅ 延迟极低（< 1ms）                                             │
│ ✅ 吞吐量高（不等待副本）                                        │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│ Kafka：同步/异步可配置（acks 参数）                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│ 配置1：acks=0（异步，不等待任何确认）                            │
│ ├─ 客户端发送消息后立即返回                                      │
│ ├─ 不等待 Leader 写入                                            │
│ ├─ 延迟：< 1ms                                                   │
│ └─ 风险：可能丢失消息                                            │
│                                                                  │
│ 配置2：acks=1（等待 Leader 确认）                                │
│ ├─ 客户端等待 Leader 写入磁盘                                    │
│ ├─ 不等待 Follower 同步                                          │
│ ├─ 延迟：1-5ms                                                   │
│ └─ 风险：Leader 宕机前未同步到 Follower，数据丢失               │
│                                                                  │
│ 配置3：acks=all（等待所有 ISR 副本确认）⭐推荐                   │
│                                                                  │
│ 客户端写入流程：                                                  │
│                                                                  │
│ 1. 客户端发送消息到 Partition 0                                  │
│    └─ 客户端根据 key 的 hash 路由到 Partition                    │
│                                                                  │
│ 2. Leader (Broker1) 处理：                                       │
│    ├─ 写入磁盘（Page Cache + fsync）                            │
│    ├─ 等待所有 ISR Follower 确认 ⏳                              │
│    └─ 所有 ISR 确认后，返回 OK 给客户端 ✅                       │
│                                                                  │
│ 3. Follower (Broker2, Broker3) 处理：                            │
│    ├─ 从 Leader 拉取消息（Fetch Request）                       │
│    ├─ 写入磁盘                                                   │
│    └─ 发送 ACK 给 Leader                                         │
│                                                                  │
│ 4. Leader 收到所有 ISR 的 ACK：                                  │
│    └─ 返回 OK 给客户端                                           │
│                                                                  │
│ 时间线：                                                          │
│ T0: 客户端发送消息                                               │
│ T1: Leader 写入磁盘（2ms）                                       │
│ T2: Follower1 拉取并写入（3ms）                                  │
│ T3: Follower2 拉取并写入（3ms）                                  │
│ T4: Leader 收到所有 ACK                                          │
│ T5: Leader 返回 OK（客户端收到响应）✅                           │
│                                                                  │
│ 优势：                                                            │
│ ✅ 数据不丢失（所有 ISR 都有数据）                               │
│ ✅ 强一致性保证                                                  │
│ ✅ 可配置（根据业务需求选择 acks）                               │
│                                                                  │
│ 劣势：                                                            │
│ ❌ 延迟较高（5-10ms）                                            │
│ ❌ 吞吐量降低（等待副本确认）                                    │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘

性能对比（单次写入）：

| 配置 | Redis Cluster | Kafka (acks=0) | Kafka (acks=1) | Kafka (acks=all) |
|------|---------------|----------------|----------------|------------------|
| **延迟** | 0.5-1ms | 0.5-1ms | 1-5ms | 5-10ms |
| **数据丢失风险** | 高 | 极高 | 中 | 极低 |
| **吞吐量** | 10万+ QPS | 100万+ TPS | 50万+ TPS | 10万+ TPS |
| **适用场景** | 缓存、会话 | 日志收集 | 一般消息 | 金融、订单 |
```

**2. ISR 机制详解（Kafka 独有）**

```
ISR（In-Sync Replicas）：与 Leader 保持同步的副本集合

┌─────────────────────────────────────────────────────────────────┐
│ ISR 机制工作原理                                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│ Partition 0 的副本：                                             │
│ ├─ Leader: Broker1                                              │
│ ├─ Follower1: Broker2（在 ISR 中）                              │
│ └─ Follower2: Broker3（在 ISR 中）                              │
│                                                                  │
│ ISR 列表：[Broker1, Broker2, Broker3]                           │
│                                                                  │
│ 场景1：Follower2 网络延迟，落后 Leader 太多                      │
│ ├─ Leader 检测到 Follower2 延迟 > replica.lag.time.max.ms      │
│ ├─ 将 Follower2 从 ISR 中移除                                   │
│ └─ ISR 列表：[Broker1, Broker2]                                 │
│                                                                  │
│ 场景2：Follower2 恢复，追上 Leader                               │
│ ├─ Follower2 追上 Leader 的 offset                              │
│ ├─ Leader 将 Follower2 加回 ISR                                 │
│ └─ ISR 列表：[Broker1, Broker2, Broker3]                        │
│                                                                  │
│ 场景3：Leader 宕机，选举新 Leader                                │
│ ├─ 只能从 ISR 中选举新 Leader                                    │
│ ├─ 保证新 Leader 有完整数据                                      │
│ └─ 不会从非 ISR 副本中选举（避免数据丢失）                       │
│                                                                  │
│ 配置参数：                                                        │
│ replica.lag.time.max.ms=10000  # Follower 落后 10 秒移出 ISR    │
│ min.insync.replicas=2          # 最少 ISR 副本数（推荐）         │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘

Redis Cluster 没有 ISR 机制：
❌ 无法保证 Slave 与 Master 同步
❌ 无法配置最少副本数
❌ Master 宕机时，Slave 可能没有最新数据
```

**3. 故障转移对比**

```
┌─────────────────────────────────────────────────────────────────┐
│ Redis Cluster 故障转移（Raft 选举）                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│ 场景：Master 1 宕机                                              │
│                                                                  │
│ 1. 检测阶段（5-15 秒）                                           │
│    ├─ 其他节点通过 Gossip 协议检测 Master 1 下线                │
│    ├─ 半数以上节点标记 Master 1 为 PFAIL（主观下线）            │
│    └─ 达到半数后，标记为 FAIL（客观下线）                        │
│                                                                  │
│ 2. 选举阶段（5-10 秒）                                           │
│    ├─ Slave 1 发起选举                                           │
│    ├─ 向所有 Master 节点请求投票                                 │
│    ├─ 获得半数以上投票后，提升为 Master                          │
│    └─ 通过 Gossip 协议广播新的拓扑                               │
│                                                                  │
│ 3. 数据一致性                                                    │
│    ├─ Slave 1 可能没有 Master 1 的最新数据                      │
│    ├─ 丢失 Master 1 宕机前未复制的数据                           │
│    └─ 无法保证数据完整性 ❌                                      │
│                                                                  │
│ 总耗时：10-30 秒                                                 │
│ 数据丢失：可能丢失最后几秒的数据                                  │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│ Kafka 故障转移（ZooKeeper/KRaft 选举）                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│ 场景：Leader (Broker1) 宕机                                      │
│                                                                  │
│ 1. 检测阶段（1-3 秒）                                            │
│    ├─ ZooKeeper 检测到 Broker1 的心跳超时                       │
│    ├─ 标记 Broker1 为下线                                        │
│    └─ 触发 Controller 进行 Leader 选举                           │
│                                                                  │
│ 2. 选举阶段（1-2 秒）                                            │
│    ├─ Controller 从 ISR 列表中选举新 Leader                      │
│    ├─ 优先选择 ISR 中 offset 最大的 Follower                     │
│    ├─ 更新元数据到 ZooKeeper                                     │
│    └─ 通知所有 Broker 新的 Leader 信息                           │
│                                                                  │
│ 3. 数据一致性                                                    │
│    ├─ 新 Leader 一定在 ISR 中                                    │
│    ├─ 新 Leader 有完整的数据（acks=all 保证）                   │
│    └─ 不会丢失数据 ✅                                            │
│                                                                  │
│ 总耗时：2-5 秒                                                   │
│ 数据丢失：0（如果配置 acks=all + min.insync.replicas=2）        │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘

对比总结：

| 维度 | Redis Cluster | Kafka |
|------|---------------|-------|
| **检测时间** | 5-15 秒（Gossip 协议） | 1-3 秒（ZooKeeper 心跳） |
| **选举时间** | 5-10 秒（Raft 选举） | 1-2 秒（Controller 选举） |
| **总耗时** | 10-30 秒 | 2-5 秒 |
| **数据丢失** | 可能丢失（异步复制） | 不丢失（acks=all + ISR） |
| **选举条件** | 任意 Slave 可参选 | 只能从 ISR 中选举 |
| **一致性保证** | 最终一致性 | 强一致性（可配置） |
```

**4. 读写模式对比**

```
┌─────────────────────────────────────────────────────────────────┐
│ Redis Cluster：主读主写（默认）                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│ 默认模式：                                                        │
│ ├─ 读：只从 Master 读取                                          │
│ ├─ 写：只写 Master                                               │
│ └─ Slave：只做备份，不提供读服务                                 │
│                                                                  │
│ 读写分离模式（需要配置）：                                        │
│ ├─ 客户端执行：READONLY 命令                                     │
│ ├─ 读：可以从 Slave 读取                                         │
│ ├─ 写：仍然只写 Master                                           │
│ └─ 风险：可能读到延迟数据                                        │
│                                                                  │
│ Java 配置（Lettuce）：                                           │
│ LettuceClientConfiguration config = LettuceClientConfiguration  │
│     .builder()                                                   │
│     .readFrom(ReadFrom.REPLICA_PREFERRED)  # 优先读 Slave       │
│     .build();                                                    │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│ Kafka：Leader 读写（强制）                                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│ 强制模式：                                                        │
│ ├─ 读：只能从 Leader 读取                                        │
│ ├─ 写：只能写 Leader                                             │
│ └─ Follower：只做同步复制，不提供读写服务                        │
│                                                                  │
│ 原因：                                                            │
│ ├─ 保证强一致性（所有消费者读到相同数据）                        │
│ ├─ 简化架构（不需要处理读写分离的复杂性）                        │
│ └─ Kafka 设计目标是高吞吐，单 Leader 足够                        │
│                                                                  │
│ 注意：                                                            │
│ ❌ 无法配置从 Follower 读取                                      │
│ ❌ 无法实现读写分离                                              │
│ ✅ 但 Kafka 的 Leader 性能足够高（百万 TPS）                     │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

#### 🎯 适用场景对比

**Redis Cluster 适用场景：**

```
✅ 高性能缓存
   - 商品详情缓存（QPS > 10万）
   - 用户会话存储
   - 热点数据缓存

✅ 低延迟 KV 存储
   - 延迟要求 < 1ms
   - 实时排行榜
   - 计数器、限流

✅ 简单数据结构
   - String、Hash、List、Set、ZSet
   - 不需要复杂查询

✅ 可容忍少量数据丢失
   - 缓存数据可重建
   - 会话数据可重新登录

示例代码：
// Redis Cluster 用于商品缓存
@Service
public class ProductService {
    @Autowired
    private RedisTemplate<String, Product> redisTemplate;
    
    public Product getProduct(Long productId) {
        String key = "product:" + productId;
        Product product = redisTemplate.opsForValue().get(key);
        
        if (product == null) {
            product = productMapper.selectById(productId);
            redisTemplate.opsForValue().set(key, product, 1, TimeUnit.HOURS);
        }
        
        return product;
    }
}
```

**Kafka 适用场景：**

```
✅ 消息队列
   - 异步解耦（订单 → 库存 → 物流）
   - 削峰填谷（秒杀场景）
   - 事件驱动架构

✅ 日志收集
   - 应用日志聚合
   - 用户行为日志
   - 系统监控指标

✅ 流处理
   - 实时计算（Kafka Streams）
   - 数据管道（ETL）
   - CDC（Change Data Capture）

✅ 数据不能丢失
   - 订单消息
   - 支付消息
   - 金融交易

示例代码：
// Kafka 用于订单消息
@Service
public class OrderService {
    @Autowired
    private KafkaTemplate<String, Order> kafkaTemplate;
    
    @Transactional
    public void createOrder(Order order) {
        // 1. 保存订单到数据库
        orderMapper.insert(order);
        
        // 2. 发送订单消息到 Kafka
        kafkaTemplate.send("order-topic", order.getId().toString(), order);
        
        // 3. Kafka 保证消息不丢失（acks=all）
    }
}

// 消费者：库存服务
@KafkaListener(topics = "order-topic")
public void handleOrder(Order order) {
    // 扣减库存
    inventoryService.deduct(order.getProductId(), order.getQuantity());
}
```

#### 💡 选择建议

**选择 Redis Cluster：**

```
场景特征：
✅ 需要极低延迟（< 1ms）
✅ 高并发随机读写
✅ 数据可丢失或可重建
✅ 简单的 KV 存储
✅ 需要丰富的数据结构

典型应用：
- 电商商品缓存
- 用户会话管理
- 实时排行榜
- 分布式锁
- 限流计数器

不适合：
❌ 消息队列（顺序消费、持久化）
❌ 日志收集（大量数据、长期存储）
❌ 数据不能丢失的场景
```

**选择 Kafka：**

```
场景特征：
✅ 消息队列、事件流
✅ 数据不能丢失
✅ 需要顺序消费
✅ 大量数据写入（日志、埋点）
✅ 流处理、实时计算

典型应用：
- 订单消息队列
- 用户行为日志收集
- 微服务异步通信
- CDC（数据库变更捕获）
- 实时数据管道

不适合：
❌ 低延迟 KV 查询（< 1ms）
❌ 随机读写
❌ 复杂数据结构（Hash、List、Set）
❌ 缓存场景
```

**混合使用（推荐）：**

```
架构：Redis Cluster（缓存） + Kafka（消息队列） + MySQL（持久化）

典型流程：
1. 用户下单 → 写入 MySQL（持久化）
2. 发送订单消息到 Kafka（异步解耦）
3. 订单详情缓存到 Redis（高性能查询）
4. 库存服务消费 Kafka 消息（扣减库存）
5. 物流服务消费 Kafka 消息（发货）

优势：
✅ Redis 提供低延迟查询（< 1ms）
✅ Kafka 保证消息不丢失（acks=all）
✅ MySQL 保证数据持久化（ACID）
✅ 各司其职，发挥各自优势

示例代码：
@Service
public class OrderService {
    @Autowired
    private RedisTemplate redisTemplate;
    
    @Autowired
    private KafkaTemplate kafkaTemplate;
    
    @Autowired
    private OrderMapper orderMapper;
    
    @Transactional
    public void createOrder(Order order) {
        // 1. 持久化到 MySQL
        orderMapper.insert(order);
        
        // 2. 发送消息到 Kafka（异步处理）
        kafkaTemplate.send("order-topic", order);
        
        // 3. 缓存到 Redis（快速查询）
        String key = "order:" + order.getId();
        redisTemplate.opsForValue().set(key, order, 1, TimeUnit.HOURS);
    }
    
    public Order getOrder(Long orderId) {
        // 先查 Redis
        String key = "order:" + orderId;
        Order order = (Order) redisTemplate.opsForValue().get(key);
        
        if (order == null) {
            // 缓存未命中，查 MySQL
            order = orderMapper.selectById(orderId);
            redisTemplate.opsForValue().set(key, order, 1, TimeUnit.HOURS);
        }
        
        return order;
    }
}
```

## 面试重点

**Q1：RDB 和 AOF 的区别？**

| 特性 | RDB | AOF |
|------|-----|-----|
| 文件大小 | 小 | 大 |
| 恢复速度 | 快 | 慢 |
| 数据安全 | 低 | 高 |
| 性能影响 | 小 | 稍大 |

**Q2：Redis 如何保证高可用？**

三种方案：
1. **主从复制**：读写分离，提高读性能
2. **哨兵**：自动故障转移
3. **Cluster**：数据分片 + 高可用

**Q3：Cluster 如何实现数据分片？**

- 16384 个 slot
- CRC16(key) % 16384 计算 slot
- 支持 Hash Tag 控制分片
